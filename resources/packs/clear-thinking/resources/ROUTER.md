# clear-thinking 路由器

本文件只负责把当前问题路由到：判断缺口、动作类型、组合路径、停止/升级条件。具体微技能只通过 `MICRO_SKILLS.md` 定位；采用前必须读取对应 `micro-skills/<微技能>.md`。

## 1. 路由硬规则

1. 先判断缺口类型，不按“产品/创业/生活/某材料”等关键词套模板。
2. 先选动作类型，再去 `MICRO_SKILLS.md` 找具体文件。
3. 只看索引只能得到“候选”；声明采用前必须读取具体微技能文件。
4. 达到可行动、需工具、需追问、需子代理或需 Workflow 条件时，停止继续分析。
5. 默认读取 1-5 个具体微技能；复杂任务最多 10 个；仍无法裁剪时拆子代理或 Workflow。

## 2. 路由流程

```text
感知输入 → 判断缺口 → 选择动作类型 → 查 MICRO_SKILLS.md → 读取具体微技能 → 组合输出 → 停止/升级
```

### 2.1 感知输入

```markdown
- 用户表层请求是什么？
- 用户真正想改变的状态是什么？
- 已知事实有哪些？哪些只是推断或假设？
- 当前硬约束、风险和不可逆动作是什么？
- 本轮是否要修改文件、调研、复盘或完成检查？
- 是否需要先查证、追问或使用工具？
```

### 2.2 缺口判断表

| 缺口/动作类型 | 判断信号 | 下一步 |
| --- | --- | --- |
| 事实校准 | 事实、推断、样本、信息缺口不清 | 到 `MICRO_SKILLS.md` 筛选“事实校准” |
| 目标取舍 | 目标、代价、优先级、不可兼得项不清 | 筛选“目标取舍” |
| 问题诊断 | 问题很多，但不知道卡在哪一层 | 筛选“问题诊断” |
| 风险边界 | 高成本、不可逆、危机、责任边界不清 | 筛选“风险边界” |
| 结构治理 | 多人、多钱、多权责、多机制承接不清 | 筛选“结构治理” |
| 产品表达 | 需求、页面、字段、体验需要表达清楚 | 筛选“产品表达” |
| 增长信任 | 用户、口碑、可信度、长期资产不清 | 筛选“增长信任” |
| 执行验证 | 已有判断，但缺行动、验收或复盘 | 筛选“执行验证” |
| 关系承接 | 关系、人情、外部帮助、主体责任不清 | 筛选“关系承接” |
| 方法建设 | 需要从材料中提炼、封装或去权威背书 | 筛选“方法建设” |

读取提示：

- 不用“跨几个目录”判断任务大小；看材料量、阶段数、验证成本、交叉审查需求和闭环回写需求。
- `CHECKLISTS.md` 只在完成前检查、用户要求“别浅/别教条/检查是否靠谱”或高风险决策时读取。
- 一旦满足子代理或 Workflow 条件，不继续在主会话读取更多微技能来硬撑。

## 3. 微技能选择

选择时内部回答：

```markdown
- 本轮最关键的 1 个主要矛盾是什么？
- 这个矛盾缺哪 1-3 类判断动作？
- 每类动作最少需要哪个微技能？
- 哪些看似相关但本轮不用？为什么？
- 如果判断错了，最早会出现什么失效信号？
```

数量控制：

| 任务复杂度 | 微技能数量 | 处理方式 |
| --- | --- | --- |
| 低 | 0-2 个 | 内部想清楚后直接执行或短答 |
| 中 | 2-4 个 | 标准组合，输出主要判断和行动 |
| 高 | 5 个以内 | 先裁剪为最小组合；材料复杂时交给子代理 |
| 超高 | 6 个以上候选 | 不在主会话硬读，拆子代理或 Workflow |

## 4. 组合模式

### 4.1 轻量组合

```text
目标取舍 / 事实校准 → 执行验证
```

适用：低风险、目标基本明确、材料少。输出只保留当前判断、下一步、验收方式。

### 4.2 标准组合

```text
事实校准 → 问题诊断 → 1-2 个关键动作类型 → 执行验证
```

要求：只选 1 个主要矛盾，明确暂不处理项。

### 4.3 深度组合

```text
事实校准 → 目标取舍 → 问题诊断 → 结构治理 / 风险边界 / 关系承接 → 执行验证
```

适用：多目标、多角色、多后果、用户要求“不要浅”、需要沉淀方法或技能。深度组合仍优先裁剪到 5 个以内。

### 4.4 典型组合模板

| 问题形态 | 先判断 | 可选补充 | 停止条件 |
| --- | --- | --- | --- |
| 模糊合作、帮忙、转介、担保 | 关系边界 | 涉及钱、名义责任或退出机制时补结构治理/风险边界 | 责任、成本、退出和可拒绝事项明确 |
| 产品或 PRD 不清 | 需求真实性和表达边界 | 事实不足补事实校准；交付前补质量/完成检查 | 字段、状态、操作、验收明确 |
| B 端/组织需求提出者与使用者不一致 | 诉求来源验证分流 | 补需求四问、现场样本或相关方地图 | 验证对象、真实使用者、受影响者和处理策略明确 |
| 多项目/多租户/中台需求差异是否要配置 | 配置化分流判断 | 补复用/扩展/新建、默认值制度、字段/状态/操作卡 | 写死/配置/规则化/暂缓及配置权限、默认值、回滚明确 |
| 模块、菜单、章节或职责边界不清 | 模块一句话定位校验 | 补需求四问、复用/扩展/新建、页面范式选择 | 定位句、相邻边界和保留/合并/改名/取消建议明确 |
| 准备让 AI 或团队直接生成原型、文档、演示稿、代码结构 | 前置决策校验 | 补目标澄清、模块定位、信息分档；生成后补质量裁决 | 前置决策、checkpoint、放行条件和停问信号明确 |
| 创业、副业、职业选择 | 趋势与代价 | 资源弱时补定位；合伙时补控制权；高收益时补风险 | 是否投入、投入多少、何时止损明确 |
| 组织或平台机制设计 | 目标层级和不可兼得项 | 责任状态、默认值、承接链或相关方地图 | 首个机制变更和反馈指标明确 |
| 长期项目落地 | 现场样本 | 多方承接、试点推广、阶段管理按缺口补充 | 首个试点、回流指标、暂停条件明确 |
| 高反思但难行动 | 痛感分流或解释关闭 | 涉及安全/基本保障先切风险边界 | 下一步可做动作和失败信号明确 |
| 自述、生日/年度复盘、日常随笔材料很长 | 自我叙事校准 | 强情绪先痛感分流；阶段变化补阶段轮廓；行动收口补最小行动 | 叙事事实、当前版本、保留/停用项和下一步验证明确 |
| 方法蒸馏或技能建设 | 判断动作提取 | 封装、反背书、完成检查 | 运行层动作可独立调用，不靠来源背书 |
| 复杂系统或新产品起步 | 定位四问或最小杠杆切口 | 核心用户共创闭环、爆品聚焦、完成检查 | 第一切口、暂不做清单、验证信号明确 |
| 口碑、满意度、投诉和热度信号冲突 | 口碑交叉验证 | 用户朋友、现场样本校准、口碑飞轮 | 多源信号置信度和优先修复项明确 |
| 重大新业务、第二曲线或跨行业进入 | 顺势判断 | 重大进入资源底盘核验、长期复利、阶段轮廓 | 进入/试点/观察/放弃和阶段门明确 |
| 增长失速或规模化后质量/交付/用户连接下滑 | 失速后的能力缺口治理 | 技术-规则-价值三层诊断、效率革命、口碑交叉验证 | 1-3 个能力缺口、负责人和验证指标明确 |
| 长期用户承诺、平台规则、价格/隐私底线 | 可监督承诺边界 | 约束与风险扫描、长期复利、用户朋友 | 指标口径、制度载体、监督方式和复核周期明确 |
| 情绪型产品、焦虑营销或安全陪伴类服务爆火 | 情绪安慰剂拆分 | 用户朋友、需求四问、可监督承诺边界 | 情绪慰藉、真实帮助、承接机制和承诺降级边界明确 |
| 政策、流程或平台规则目标正确但弱承接方成本不清 | 制度成本承接力校验 | 相关方损益与承接力地图、约束与风险扫描、样本试点回流循环 | 成本转嫁链、最弱承接方、缓冲方案和不可让渡底线明确 |
| 团队喊快但质量、风险或学习效果不清 | 快慢分层决策 | 敏捷债务、质量裁决、核心用户共创闭环 | 哪些环节提速、哪些不可压缩、回滚条件明确 |
| 产品功能多但用户无感，或“极致”变成内部审美 | 极致体验锚点 | 用户朋友、体验锚点校准、爆品条件校验 | 可感知体验锚点、打磨清单和成本边界明确 |
| 准备重押产品/服务/内容，希望形成突破 | 爆品条件校验 | 最小杠杆切口、极致体验锚点、口碑飞轮 | 强需求、体验差、供给、传播和持续性是否成立 |
| 成本高、交付慢、渠道重、反馈慢但瓶颈不清 | 效率瓶颈拆解 | 技术-规则-价值三层诊断、新渠道效率校验、最小行动 | 最大摩擦点、降摩擦动作和副作用边界明确 |
| 平台/中台/集团要赋能伙伴或生态团队 | 平台赋能边界 | 生态协同、承接链结构设计、相关方地图 | 平台提供、伙伴自担、质量门和退出信号明确 |
| 技术投入短期看不到收益或长期叙事过重 | 技术投入耐心账本 | 长期复利、阶段轮廓、质量裁决 | 技术投入分类、评价周期、复用资产和止损线明确 |
| 直营、代理、门店数字化、线上线下融合或渠道改造 | 新渠道效率校验 | 效率瓶颈拆解、平台赋能边界、样本试点回流循环 | 渠道链路改善、成本体验平衡和试点条件明确 |
| 旧方法、旧流程、旧原则在新阶段失灵 | 方法论版本化复盘 | 判断动作提取、反权威背书、完成检查 | 保留/升级/停用项和新版本触发条件明确 |

### 4.5 并行组合

当任务可拆为互不依赖的专项时，优先子代理并行：

```text
材料深读 + 外部资料研究 + 现有结构审查 + 风险审查 → 主会话整合
```

### 4.6 迭代组合

```text
假设 → 最小行动/实验 → 反馈 → 修正 → 下一轮
```

每轮必须回答：当前假设、最小验证、成功信号、失败信号、下一轮调整。

## 5. 特殊缺口处理

| 缺口类型 | 处理方式 |
| --- | --- |
| 目标缺口 | 用目标取舍类微技能重写目标；若仍依赖用户偏好，再追问 |
| 事实缺口 | 先查工作区索引、命中记录、相关文件或公开资料 |
| 历史缺口 | 先定位最小相关历史材料，只读会改变本轮判断的部分 |
| 材料缺口 | 子代理读取最小材料并返回摘要，主会话不吞长上下文 |
| 结构缺口 | 先诊断层级，再选结构治理类技能，不直接给建议清单 |
| 取舍缺口 | 明确不选什么、代价是什么、失败信号是什么 |
| 执行缺口 | 生成 3-7 步以内的最小行动和验收方式 |
| 验证缺口 | 使用完成检查或专项验收微技能 |

## 6. 停止与升级

### 6.1 可以停止分析并行动

- 目标已经明确；
- 下一步行动明确；
- 风险可接受；
- 验收方式明确；
- 信息缺口不会改变本轮行动。

### 6.2 需要工具

- 文件事实可查；
- 需要外部最新资料；
- 需要运行结果、截图或页面状态；
- 需要验证路径是否存在。

### 6.3 需要子代理

- 需要读取 2 份以上长材料；
- 需要并行研究多个信息源；
- 需要独立视角审查；
- 分析会显著膨胀主会话上下文。

### 6.4 需要 Workflow

- 跨阶段、跨项目、多产物；
- 需要“分析 → 执行 → 验证 → 回写”的闭环；
- 单个会话难以稳定协调；
- 需要多个子代理交叉检查。

### 6.5 需要追问用户

只有满足以下条件才追问：

- 缺口只有用户能回答；
- 继续假设会明显偏离用户真实意图；
- 方案取舍依赖用户价值偏好；
- 涉及不可逆或对外发布动作。
