# 生态协同

> 动作类型：结构治理
> 场景标签：经营、增长、创业、长期判断、工作、生活

## 适用处境

多团队、多伙伴、多产品协作时，需要判断哪些必须自研，哪些可以协作，哪些可以外包，并建立质量、接口和退出机制。

## 输入

- 共同用户、共同目标、共同标准。
- 各方能力、边界、接口、质量门槛。
- 核心资产、可替换部分、退出成本。

## 操作步骤

1. 明确共同服务的用户和共同结果；若外部压力或窗口变化，先重画共同目标和最低边界。
2. 拆分核心自研、伙伴协作、可外包/可替换部分。
3. 为协作部分定义接口、输入、输出、质量门槛和验收方式。
4. 判断哪些能力不能外包：核心判断、用户信任、关键数据、长期资产。
5. 写清不可合作对象、退出信号和自身核心能力，不用合作愿景替代边界。
6. 设计退出机制：替换路径、数据交接、责任收口、失败兜底。
7. 输出协同边界和质量控制方案。

## 关键自问

- 我们共享的是用户价值，还是只是在共享资源？
- 哪些能力一旦外包会失去长期控制权？
- 对方交付失败时，谁对用户负责？
- 接口是否清楚到可以验收，而不是靠关系协调？
- 退出时数据、责任、用户体验如何承接？

## 输出格式

```markdown
- 协同对象：
- 共同用户：
- 共同标准：
- 核心自研：
- 伙伴协作：
- 可外包 / 可替换：
- 接口与验收：
- 退出机制：
- 主要风险：
```

## 常见误用

- 把资源互换误认为生态协同。
- 因为短期省事，把核心能力交出去。
- 只谈合作愿景，不写接口和验收。
- 没有退出机制就深度绑定。

## 失效信号

单团队闭环即可解决；协作对象不共享用户价值；问题核心是内部目标不清而非生态协同。

## 常组合

- `相关方损益与承接力地图`：合作前先判断各方损益、承接条件和风险边界。
- `样本试点回流循环`：生态协同需要先用样本点位验证接口和反馈机制时。
- `资本控制权校准`：处理外部资源和控制权关系。
- `承接链结构设计`：设计多方承接责任链。
- `长期复利`：判断协作是否沉淀长期资产。
- `完成检查`：形成协作验收清单。
