# 信息一二三档

> 动作类型：产品表达
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

页面、表单、详情、列表或文档信息很多，重点不清、首屏混乱，用户不知道先看什么、再判断什么、最后在哪里行动。

## 输入

- 用户任务、角色、字段、状态、提示、操作。
- 决策顺序、使用频率、风险信息和异常说明。
- 当前页面或文档承载的主任务。

## 操作步骤

1. 明确页面或文档的主任务。
2. 列出所有信息项、字段、提示、操作、状态。
3. 按用户决策顺序分档：一档必看、二档辅助、三档低频。
4. 设计显隐方式：首屏、折叠、悬浮、详情、二级页、筛选项。
5. 检查一档是否过载，三档是否被误放到主流程。
6. 按角色差异调整信息层级。

## 关键自问

- 用户来到这里第一件要完成什么？
- 哪些信息影响“能不能做决定”？
- 哪些信息只是让人放心，但不该抢占主视线？
- 低频信息是否有可靠入口？
- 不同角色的一档是否不同？

## 输出格式

```markdown
- 主任务：
- 一档信息：
  - 字段 / 内容：
  - 理由：
  - 展示位置：
- 二档信息：
  - 展示方式：
- 三档信息：
  - 收纳方式：
- 移除 / 延后展示项：
- 验收方式：
```

## 常见误用

- 把重要信息等同于一档信息。
- 把领导关心的信息放到用户主任务前面。
- 只做视觉层级，不处理信息决策顺序。
- 把所有异常、规则说明都堆在首屏。

## 失效信号

问题不是页面、文档或信息呈现，而是业务规则不清；用户任务未定义；页面本身不该承载这么多信息，需要拆流程。

## 常组合

- `页面范式选择`：先定容器再定层级。
- `字段-状态-操作卡`：明确字段展示规则。
- `用户朋友`：防止站在内部视角堆信息。
- `效率革命`：减少查找和返工成本。
