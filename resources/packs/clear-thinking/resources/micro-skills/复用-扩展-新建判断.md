# 复用 / 扩展 / 新建判断

> 动作类型：产品表达
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

要新增功能、页面、接口或流程，但不确定是否已有能力可复用、是否应扩展旧模块，还是需要新建独立对象。

## 输入

- 现有模块、页面、接口、字段、状态和流程。
- 新需求差异、未来扩展可能、用户理解成本。
- 研发成本、维护成本、配置能力和历史约束。

## 操作步骤

1. 列出现有能力：模块、页面、字段、接口、流程、配置项。
2. 拆需求差异：对象、规则、角色、频率、风险、生命周期是否不同。
3. 判断三类路径：复用、扩展、新建。
4. 标出被复用对象、需扩展点、不可复用原因和配置项。
5. 说明不采用其他路径的理由。
6. 找一个真实样例跑通，验证建设方式不会污染旧模块或过度新建。

## 关键自问

- 这是同一个对象的不同视图，还是新对象？
- 差异是配置差异、规则差异，还是责任差异？
- 复用会不会把旧模块拖成万能泥球？
- 新建是不是只是因为不熟悉已有能力？
- 未来第二个类似需求出现时，当前方案能否承接？

## 输出格式

```markdown

```

## 常见误用

- 把“看起来像”当成可复用，忽略状态和责任差异。
- 为省工期强行复用，导致旧模块规则污染。
- 为追求干净架构过早新建。
- 只问研发成本，不问用户理解成本和运营维护成本。

## 失效信号

完全没有可比较的旧对象；当前仍在方向探索，尚未定义对象和边界；争议核心不是建设方式，而是目标是否成立。

## 常组合

- `页面范式选择`：判断是否已有成熟内部范式可延用。
- `配置化分流判断`：扩展路径涉及多项目、多租户或未来差异时，细化写死/配置/规则化边界。
- `模块一句话定位校验`：争议是新模块还是旧模块扩展时，先校验模块边界。
- `字段-状态-操作卡`：细化扩展边界。
- `长期复利`：判断复用是否形成资产而非债务。
- `敏捷债务`：防止短期复用积累结构问题。
