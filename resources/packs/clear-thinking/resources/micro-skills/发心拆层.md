# 发心拆层

> 动作类型：目标取舍
> 场景标签：AI、组织、产品、协作、工作

## 适用处境

项目同时承载用户价值、组织战役、商业指标、技术展示、领导表达等多层目标；如果不拆清主发心，需求、排期和验收会互相拉扯。

## 输入

- 项目目标、相关方、指标、场景。
- 用户、产品、组织、商业、技术等不同层面的诉求。
- 当前阶段和必须交付的结果。

## 操作步骤

1. 收集项目表层目标和相关方。
2. 拆五层发心：用户发心、产品发心、组织发心、商业发心、技术发心。
3. 标注每层目标对应的指标、受益方和成本承担方。
4. 判断当前阶段主发心：哪个目标失败，项目就算失败。
5. 明确降级目标：哪些可以顺带做，哪些暂不承诺，哪些不能牺牲。
6. 输出对需求范围、排期、验收和沟通口径的影响。

## 关键自问

- 这个项目表面说为用户，实际上还在服务谁？
- 哪个目标如果失败，项目就算失败？
- 哪些目标只是加分项，不应主导方案？
- 当前阶段是验证价值、争取资源、完成交付，还是建立能力？
- 有没有目标被包装成用户价值，但实际主要服务组织表达？
- 验收标准是否匹配主发心？

## 输出格式

```markdown
## 发心拆层
- 用户发心：
- 产品发心：
- 组织发心：
- 商业发心：
- 技术发心：

## 当前主发心
- 主发心：
- 选择理由：

## 降级目标
- 暂不承诺：
- 可顺带满足：
- 不能牺牲：

## 对执行的影响
- 需求范围：
- 排期：
- 验收：
```

## 常见误用

- 把发心拆层变成动机审判。
- 只拆目标，不选主发心。
- 所有发心都想满足，最后没有取舍。
- 把组织/商业/技术发心默认视为不纯。
- 不把主发心反映到验收标准。

## 失效信号

目标单一且无组织/商业/技术冲突；拆完后不影响范围、排期、验收；用户只需要单点执行；讨论陷入谁对谁错，而不是当前阶段服务什么。

## 最高风险边界

不要用“发心”语言给相关方贴道德标签；它只能用于拆目标来源和取舍依据，不能用于推断人格、忠诚度或立场。

## 常组合

- `不可能三角识别`：多发心冲突时取舍。
- `需求四问`：用户发心不清时回到真实场景。
- `质量裁决`：按主发心设定验收。
- `定位四问`：主发心确定后收窄服务对象。
- `组织叙事校准`：组织目标过强时校准表达和实际动作。
