# 配置化分流判断

> 动作类型：结构治理
> 场景标签：产品、SaaS、中台、后台系统、研发、工作

## 适用处境

多项目、多租户、多客户、多角色或多地区共用同一套系统/流程时，某个需求看似只是当前项目的细节，但未来可能在其他项目不同；需要判断是写死、做成配置、抽象成规则，还是暂缓。

## 不适用处境

- 一次性活动页、临时脚本、小范围单次交付，后续明确不会复用。
- 当前差异还没被验证，只是“以后可能会有很多变化”的泛化担忧。
- 配置成本、解释成本、权限成本明显高于变化收益。
- 当前争议核心是目标是否成立，而不是实现方式是否可配置。

## 输入

- 当前诉求、适用项目/租户/客户/角色。
- 已知差异点、未来变化概率、变化频率和影响范围。
- 默认值、配置权限、配置后谁维护、谁承担配置错误。
- 研发成本、实施成本、用户理解成本、长期维护成本。

## 操作步骤

1. 判差异范围：只当前场景不同，还是不同项目/角色/客户/时间都会不同？
2. 判变化概率：未来是否会反复改、按地区/客户分化，还是一次性稳定规则？
3. 判配置收益：配置能否减少返工、实施改代码、客户定制或流程分叉？
4. 判配置成本：配置项是否增加理解、测试、权限、默认值、培训和维护成本？
5. 选择路径：写死 / 参数配置 / 规则化配置 / 平台默认 + 局部覆盖 / 暂缓确认。
6. 明确配置边界：谁能配、配什么、默认值是什么、影响范围多大、错误如何回滚。

## 关键自问

- 这是当前项目独有，还是共用系统中的可变维度？
- 变化是用户偏好、业务规则、权限边界，还是实现细节？
- 做成配置后，谁真的会维护？维护错了谁承担后果？
- 默认值会不会替用户做高风险决定？
- 不配置的长期成本是否高于配置复杂度？

```markdown

- 差异范围：当前项目 / 多项目 / 多角色 / 多客户 / 时间变化

- 配置收益：
- 配置成本：
- 结论：写死 / 参数配置 / 规则化配置 / 默认+覆盖 / 暂缓

## 配置边界
- 配置项：
- 配置权限：
- 默认值：
- 影响范围：
- 回滚/兜底：
```

## 常见误用

- 把所有不确定都做成配置，制造复杂系统和实施负担。
- 只考虑研发复用，不考虑使用者理解和实施维护。
- 配置权限不清，导致一线或客户背负错误后果。
- 默认值静默生效，影响费用、权限、通知或承诺。
- 为了省当前工期写死，给后续多项目交付制造债务。

## 失效信号

无法说清差异范围和变化概率；配置项无人维护；配置结果影响高风险权益但没有显式确认；新增配置不能减少返工只会增加理解成本；问题其实是对象边界不清。命中时切换到 `复用 / 扩展 / 新建判断`、`默认值制度` 或 `模块一句话定位校验`。

## 最高风险边界

不能通过配置项把合规、安全、费用、隐私、权限和对外承诺责任静默转嫁给用户或实施人员；高风险默认值必须可知、可调、可退，并有回滚机制。

## 常组合

- `复用 / 扩展 / 新建判断`：先判断建设路径，再细化配置边界。
- `默认值制度`：配置涉及默认权限、通知、推荐、费用或隐私时。
- `字段-状态-操作卡`：把配置项落到字段、状态、权限和操作。
- `质量裁决`：交付前检查配置是否可理解、可测试、可回滚。
