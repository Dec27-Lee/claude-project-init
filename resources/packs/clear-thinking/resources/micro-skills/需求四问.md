# 需求四问

> 动作类型：产品表达
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

需求被写成功能、按钮、页面、颜色、入口、配置项或“灵活一点”；需要把方案表达还原为用户、场景、问题和目的。

## 输入

- 需求原话、角色、场景、触发条件。
- 用户当前做法、遇到的问题、想达到的目的。
- 已有方案、限制和必须保留的业务规则。

## 操作步骤

1. 保留需求原话，不立即否定或改写。
2. 回答四问：什么人、什么情况下、碰到什么问题、想达到什么目的。
3. 检查是否从功能诉求还原为场景诉求。
4. 识别伪需求：方案先行、角色缺失、场景缺失、目标模糊、只表达偏好。
5. 改写成需求句。
6. 如果仍缺信息，只追问最少问题，不连续盘问。

## 关键自问

- 这是用户问题，还是已经跳到解决方案？
- “谁”是否足够具体？
- “什么时候用”是否清楚？
- 用户遇到的是障碍、风险、效率、理解还是协作问题？
- 用户想达到的是完成任务、减少错误、节省时间、获得确认还是提高质量？
- 继续追问会不会拖慢？

## 输出格式

```markdown
## 需求四问
- 什么人：
- 什么情况下：
- 碰到什么问题：
- 想达到什么目的：

## 改写后的需求
- 当____在____情况下，遇到____问题时，需要____，以便____。

## 缺口
- 还缺：
- 最小追问：
```

## 常见误用

- 把四问当作固定访谈表，机械追问。
- 用户已经说清楚了仍继续问。
- 只补角色和场景，不识别目标。
- 把用户给出的方案完全否定，而不是还原其背后问题。
- 对内部工具需求强行套消费产品语言。

## 失效信号

四问已明确，继续追问会拖慢；任务只是执行已确认需求；用户当前需要质量裁决或方案取舍；四问答案不能影响方案。

## 常组合

- `定位四问`：先定位人群和场景，再写需求。
- `字段-状态-操作卡`：需求落到系统表达。
- `诉求来源验证分流`：提出者不等于使用者，需先确定验证对象和受影响方。
- `配置化分流判断`：需求涉及多项目、多租户或未来差异时。
- `页面范式选择`：需求清楚后选择页面形态。
- `质量裁决`：交付前检查需求是否被正确承接。
- `目标澄清`：需求背后目标不明时使用。
