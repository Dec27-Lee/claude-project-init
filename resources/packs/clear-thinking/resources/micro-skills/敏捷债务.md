# 敏捷债务

> 动作类型：执行验证
> 场景标签：AI、组织、产品、协作、工作

## 适用处境

高频发版、每日汇报、快速迭代但核心问题不变；团队有很多可见动作，却没有更接近真实答案或长期地基。

## 输入

- 近期迭代内容、核心假设、长期地基任务。
- 汇报节奏、上线频率、返工问题和被推迟事项。
- 用户反馈、质量指标和团队消耗。

## 操作步骤

1. 列出近期迭代、汇报、上线、优化事项。
2. 标出每项在验证什么核心假设。
3. 区分三类动作：验证核心假设、修补体验缺口、制造可见进展。
4. 找出被挤压的地基任务：架构、数据、权限、质量、研究、沉淀、自动化。
5. 判断债务风险：返工变多、问题重复、质量下降、认知分散。
6. 给出调整建议：暂停、合并、降频、补地基、设复盘点。

## 关键自问

- 我们是在更接近答案，还是只是在更频繁交付？
- 每次迭代验证了什么假设？
- 哪些重复问题说明底层结构没修？
- 哪些重要工作因为“不够可见”被推迟？
- 如果再这样迭代一个月，会产生什么债务？

## 输出格式

```markdown
- 当前迭代节奏：
- 核心假设：
- 近期动作分类：
  - 验证假设：
  - 修补体验：
  - 制造可见变化：
- 被挤压的地基：
- 债务风险：
- 调整建议：
- 复盘点：
```

## 常见误用

- 把所有快节奏都批判为债务。
- 用“敏捷债务”否定必要的小步验证。
- 只要求慢下来，不说明要补什么地基。
- 把管理汇报问题误判为产品迭代问题。

## 失效信号

当前只是一次性小修复；任务低风险、低复用、低长期影响；核心方向尚未确定，暂时需要探索性试错。

## 常组合

- `长期复利`：判断短期交付是否损害长期资产。
- `效率革命`：找返工和等待源头。
- `复用-扩展-新建判断`：判断结构债务来自强行复用还是过度新建。
- `组织叙事校准`：识别战役压力下的假进展。
