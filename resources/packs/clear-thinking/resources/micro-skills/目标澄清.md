# 目标澄清

> 动作类型：目标取舍
> 场景标签：工作、生活、组织、个人、复盘

## 适用处境

用户目标宽泛、产物不清、验收不清，或同一句请求里同时包含要建议、要执行、要判断、要安慰、要复盘等多种意图。

## 输入

- 用户原话、当前上下文、已知约束。
- 用户希望改变的状态、交付物、时间限制。
- 已知事实、待确认假设和不可触碰边界。

## 操作步骤

1. 识别表层请求：要答案、要方案、要执行、要判断、要复盘，还是要表达材料。
2. 提炼真实目标：用户真正想改变什么状态，降低什么不确定性，避免什么后果。
3. 明确本轮交付物：结论、清单、计划、草稿、检查结果或可执行动作。
4. 划定范围：本轮做什么、暂不做什么、哪些只是后续事项。
5. 判断是否追问：只有缺口会改变方向或依赖用户偏好时才追问；可合理假设推进时先说明假设后行动。
6. 若目标已明确且低风险，停止澄清，转入 `最小行动` 或直接执行。

## 关键自问

- 用户真正要的是做完一件事，还是想清楚该不该做？
- 用户要的是结果、理由、过程，还是可复用方法？
- 目标里有没有互相冲突的要求？
- 当前最不清楚的是产物、范围、验收，还是优先级？
- 哪些问题只有用户能决定，哪些可以先用默认假设推进？

## 输出格式

```markdown
## 我理解的真实目标
-

## 本轮交付
-

## 本轮范围
- 做：
- 不做：

## 默认假设
-

## 需要确认
- 仅列不确认就会改变方向的问题
```

## 常见误用

- 用户目标已经明确时仍反复追问。
- 把目标澄清变成完整需求访谈，拖慢低风险任务。
- 只复述用户原话，没有提炼真实改变状态。
- 忽略“本轮不做什么”，导致范围膨胀。
- 把自己的偏好当成用户目标。

## 失效信号

目标、产物和验收已经明确；用户明确要求不要分析、直接执行；当前问题是事实缺口或风险边界，而不是目标缺口。命中时切换到对应微技能。

## 常组合

- `事实-推断-假设分离`：目标里夹杂未经验证判断时。
- `主要矛盾`：多个目标互相冲突时。
- `方案取舍`：目标明确但路径冲突时。
- `最小行动`：目标清楚后落到下一步。
- `完成检查`：验收时回到原始目标核对。
