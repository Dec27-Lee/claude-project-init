# 诉求来源验证分流

> 动作类型：关系承接
> 场景标签：产品、B端、组织、需求、工作

## 适用处境

一个需求、建议或改动请求来自领导、业务负责人、一线人员、客户代表、实施团队或转述者，但提出者不一定是最终使用者；如果直接照做，可能把展示诉求当成一线痛点，或把局部个例当成全局需求。

## 不适用处境

- 法规、合同、安全、合规等硬性要求已经明确，只需转执行和风险控制。
- 需求来源不可得，但时间要求必须先做最小可逆处理。
- 当前讨论已经滑向动机揣测、职位评价、站队或人身标签。
- 用户只要求整理现有原话，不需要判断验证路径。

## 输入

- 诉求原话、提出者、转述链路和提出场景。
- 直接使用者、间接受益者、被影响者、决策/验收者。
- 业务目标、功能目标、权限/流程影响。
- 是否存在汇报展示、管理控制、效率提升、风险规避或一线操作痛点。

## 操作步骤

1. 保留诉求原话，先不评价提出者动机。
2. 标注来源链路：谁提出、谁转述、谁拍板、谁验收。
3. 标注使用与影响：谁每天用、谁偶尔用、谁收益、谁付出成本、谁承担风险。
4. 选择验证路径：
   - 一线高频痛点：优先现场验证任务流程和失败成本。
   - 管理/业务负责人诉求：补一线样本，确认是否真痛或只是管理口径。
   - 领导/展示诉求：区分实质目标、汇报表达和品牌/形象要求。
   - 转述诉求：回到原始使用者或最小现场样本。
5. 输出处理策略：优先解决、二次验证、低投入满足、拆分处理或暂缓。

## 关键自问

- 提出者、使用者、受影响者是不是同一批人？
- 谁最痛，谁最有话语权，谁会承担改动成本？
- 这个诉求是要解决使用问题、管理问题、展示问题，还是风险问题？
- 我是在设计验证路径，还是在揣测某类人的动机？
- 如果只听提出者，最可能漏掉哪个受影响方？

## 输出格式

```markdown
## 诉求来源
- 原话：
- 提出者 / 转述者 / 决策者：
- 直接使用者：
- 被影响者：

## 验证分流
- 诉求性质：一线痛点 / 管理流程 / 汇报展示 / 风险合规 / 转述待确认
- 最小验证对象：
- 需要补问的问题：
- 处理策略：优先解决 / 二次验证 / 低投入满足 / 拆分 / 暂缓

## 风险
- 可能被忽略的成本方：
- 不能直接下结论的部分：
```

## 常见误用

- 按职位、部门、性格或关系给需求贴好坏标签。
- 把领导诉求一概判为虚需求，或把一线诉求一概判为最高优先级。
- 为了显得“懂组织”而揣测动机，不做现场验证。
- 忽略被影响者的额外工作、权限变化和风险承担。

## 失效信号

输出开始变成人身判断、阵营划分或动机推测；无法导出验证对象和下一步；涉及劳动权益、隐私、安全、合规或高压权力关系但没有边界；受影响者无法安全表达。命中时停止分流，切换到 `约束与风险扫描`、`相关方损益与承接力地图` 或专业处理。

## 最高风险边界

不能把本技能用于歧视、排斥、压制一线、迎合权力或操控相关方；来源只影响验证路径，不决定人的价值，也不能替代事实证据。

## 常组合

- `需求四问`：把诉求还原为用户、场景、问题和目的。
- `相关方损益与承接力地图`：多方损益和承接条件复杂时。
- `现场样本校准`：需要回到真实使用现场验证时。
- `方案取舍`：多个来源诉求冲突，需要取舍时。
