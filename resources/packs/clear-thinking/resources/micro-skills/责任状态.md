# 责任状态

> 动作类型：结构治理
> 场景标签：AI、组织、产品、协作、工作

## 适用处境

已读、已处理、已触达、已确认、已完成、已推荐、已分配等状态会被他人看到，并可能影响信任、评价、协作或问责。

## 输入

- 状态触发条件、展示对象、可撤销性。
- 系统感知、用户确认、规则判定和对外展示之间的差异。
- 错误状态可能造成的关系和责任后果。

## 操作步骤

1. 列出状态词和状态出现位置。
2. 拆状态来源：系统感知、用户主动确认、管理规则判定、对外展示推断。
3. 判断展示对象和责任后果。
4. 设计状态表达强度：弱提示、待确认、明确确认、不对外展示。
5. 明确撤销、纠错、解释和日志规则。
6. 对可能问责的状态，默认要求显式确认或弱化表达。

## 关键自问

- 这个状态是事实，还是系统推断？
- 用户是否明确同意承担这个状态含义？
- 谁会看到这个状态，看到后会采取什么行动？
- 错误状态会伤害谁？
- 是否需要“未确认”而不是“已确认”？

## 输出格式

```markdown
- 状态名：
- 状态类型：系统感知 / 用户确认 / 规则判定 / 对外推断
- 触发条件：
- 展示对象：
- 责任含义：
- 可撤销 / 纠错方式：
- 推荐表达：
```

## 常见误用

- 把“系统检测到”写成“用户已确认”。
- 把“已触达”暗示成“已理解”。
- 把“已处理”暗示成“问题已解决”。
- 对外展示状态前不考虑关系成本。

## 失效信号

状态只给本人看且不产生责任后果；状态是纯技术运行状态，不影响协作或关系；当前还没有状态设计问题。

## 最高风险边界

不能把系统推断包装成用户承诺；不能用状态展示替代真实沟通；任何可能影响信任、评价、考核、问责的状态，都必须有明确触发条件、纠错机制和责任边界。

## 常组合

- `字段-状态-操作卡`：落成状态规格。
- `组件权力`：判断状态展示是否制造压力。
- `默认值制度`：处理自动确认、自动分配、自动提醒。
- `用户朋友`：保护用户信任。
