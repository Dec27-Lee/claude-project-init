# 字段 / 状态 / 操作卡

> 动作类型：产品表达
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

PRD、原型或代码实现前，需要把字段、状态、权限、操作结果和异常反馈写清楚，避免研发和测试靠猜。

## 输入

- 业务规则、数据依据、角色权限、操作结果。
- 字段来源、状态触发条件、异常和回滚规则。
- 研发、测试和运营需要验收的对象。

## 操作步骤

1. 分三类列对象：字段、状态、操作。
2. 为字段写来源、类型、展示、编辑、空值、异常和权限。
3. 为状态写触发条件、展示对象、下一状态、可撤销性和责任含义。
4. 为操作写操作人、可操作条件、成功结果、失败反馈、日志/通知。
5. 标注角色差异、边界情况和不可猜测项。
6. 输出给研发和测试可直接验收的对象卡。

## 关键自问

- 字段从哪里来，是否可信，什么时候为空？
- 状态是系统判断还是用户确认？
- 操作成功后改变了什么对象、状态或通知？
- 谁看得见，谁能操作，谁承担后果？
- 异常情况是否比正常情况更关键？

## 输出格式

```markdown
## 字段卡
- 字段名：
- 来源：
- 展示规则：
- 编辑规则：
- 空值 / 异常：
- 权限：

## 状态卡
- 状态名：
- 触发条件：
- 展示对象：
- 可撤销性：
- 责任含义：

## 操作卡
- 操作名：
- 操作人：
- 可操作条件：
- 成功结果：
- 失败反馈：
- 日志 / 通知：
```

## 常见误用

- 过早细化字段，掩盖方向未定。
- 只写正常状态，不写边界状态。
- 把权限写成角色名，不写判断条件。
- 把操作结果写成“保存成功”，不写业务对象变化。

## 失效信号

需求还在探索主目标，不该进入规格层；缺少业务规则来源，字段卡会变成猜测；争议集中在是否做，而不是怎么实现。

## 常组合

- `责任状态`：处理已读、已确认等状态风险。
- `默认值制度`：审计默认值和自动行为。
- `页面范式选择`：形成可研发实现页面规格。
- `完成检查`：作为验收清单。
