# 页面范式选择

> 动作类型：产品表达
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

列表、详情、表单、看板、树表、统计明细等页面范式不确定；页面既想展示信息又想承载操作，结构开始混乱。

## 输入

- 数据特征、操作模式、主从关系、扩展性。
- 主对象、用户主任务、内部已有页面范式。
- 批量/单条、查看/编辑、流程/指标等差异。

## 操作步骤

1. 明确主对象：记录、任务、审批、资源、关系、指标还是内容。
2. 判断用户主操作：查找、编辑、审核、分配、追踪、分析或配置。
3. 判断数据结构：集合、单体、层级、流程、指标。
4. 匹配成熟范式：列表、详情、表单、看板、树表、统计+明细等。
5. 查找内部历史页面或标准组件，优先复用用户习惯。
6. 给出范式选择和不选其他范式的理由。

## 关键自问

- 用户主要是在找、改、看、批、配、追踪还是分析？
- 数据是集合、单体、层级、流程，还是指标？
- 操作是批量优先还是单条优先？
- 页面需要承接未来扩展吗？
- 现有内部范式是否已经形成用户习惯？

## 输出格式

```markdown
- 推荐范式：
- 主对象：
- 主任务：
- 数据结构：
- 操作模式：
- 内部可参考页面：
- 不选其他范式的理由：
- 扩展预留：
```

## 常见误用

- 因为研发方便，把所有东西做成表格。
- 因为视觉好看，把流程问题做成卡片。
- 指标页面没有明细入口，导致无法解释。
- 详情页承载批量操作，列表页承载复杂编辑。

## 失效信号

当前问题不涉及界面结构；业务对象尚未定义；争议是权限、规则或数据质量，不是范式选择。

## 常组合

- `复用-扩展-新建判断`：优先找成熟内部范式。
- `信息一二三档`：确定页面内部层级。
- `字段-状态-操作卡`：落到可实现规格。
- `组件权力`：判断范式是否改变责任感和控制感。
