# 最小行动

> 动作类型：执行验证
> 场景标签：工作、生活、组织、个人、复盘

## 适用处境

已经有基本判断，需要把分析落到下一步；继续讨论会增加噪声，而不是增加确定性。

## 输入

- 目标、主要矛盾、已有事实和资源。
- 风险边界、可用时间、可接受失败成本。
- 本轮要验证或推进的对象。

## 操作步骤

1. 确认最低行动条件：目标大致明确、关键事实足够、风险可控。
2. 明确本轮动作目的：验证事实、验证偏好、验证方案，还是产出一个可用版本。
3. 把范围缩到最小：最小对象、最短反馈周期、最低不可逆成本。
4. 写清输入、动作、输出和验收标准。
5. 补一行预想结果校准：预想结果是什么、反馈指标是什么、结果不符时修正哪个判断。
6. 设定失败信号、停止条件和后手。
7. 若目标或事实仍会改变方向，先转 `目标澄清` 或 `信息缺口定位`。

## 关键自问

- 这一步是在验证，还是在正式投入？
- 能不能把范围再缩小一半？
- 输出是否可被检查，而不是只产生感觉？
- 最快什么时候能得到反馈？
- 失败会损失什么，是否可承受？
- 这一步是否命中主要矛盾，还是只是在做容易的事？

## 输出格式

```markdown

## 输入
-

## 输出

- 失败后：
```

## 常见误用

- 把完整计划伪装成最小行动。
- 只给建议，不给可执行动作。
- 行动没有验收标准，做完无法判断。
- 为了快而跳过会改变方向的关键事实。
- 选择了容易做但不命中主要矛盾的动作。

## 失效信号

目标不清、关键事实缺失、行动不可逆且未经验证，或用户要的是判断/取舍而不是马上行动。命中时先切换到对应判断技能。

## 常组合

- `主要矛盾`：确保行动打在关键处。
- `方案取舍`：推荐方案后落地第一步。
- `信息缺口定位`：把补证设计成最快反馈动作。
- `完成检查`：行动后验收。
- `目标澄清`：确认行动服务于什么目标。
