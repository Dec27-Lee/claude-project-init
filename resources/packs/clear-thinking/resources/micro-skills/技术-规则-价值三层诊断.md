# 技术-规则-价值三层诊断

> 动作类型：问题诊断
> 场景标签：工作、生活、组织、个人、复盘

## 适用处境

一个问题反复修工具、换流程、补沟通仍不稳定，怀疑不是单一技术问题，而是规则、激励或价值取舍错位。

## 不适用处境

事实未查清；单点故障已有明确修复路径；当前需要救急、安全保护或资源补齐；没有重复失败证据时，不应直接上升层级归因。

## 输入

- 问题表现、已尝试的技术修复、重复失败证据。
- 规则/激励/权责关系。
- 当事人真实目标、代价承接意愿和资源约束。

## 操作步骤

1. 先看技术层：信息、工具、能力、流程、权限、数据是否足够。
2. 再看规则层：权责、激励、制度、协作关系是否扭曲，是否让正确行为变得不经济。
3. 最后看价值层：各方是否真的愿意为目标支付代价，是否期待破格、侥幸或他人代付。
4. 找最低可改层，不随意把问题上升为文化、认知或价值观。
5. 设计验证动作，确认层级判断是否正确。
6. 若低层问题未查清，先修低层，不做宏大归因。

## 关键自问

- 这个问题换工具、补信息、改流程能否稳定解决？
- 如果规则不变，技术修复会不会失效？
- 哪个激励让人持续做出错误选择？
- 各方是否真的愿意为目标付出对应代价？
- 我是否把资源不足、制度压迫或单点故障归因为个人价值选择？

## 输出格式

```markdown
- 问题表现：
- 技术层判断：
- 规则层判断：
- 价值层判断：
- 优先修复层：
- 验证动作：
- 不上升理由：
```

## 常见误用

- 动不动上升到价值观或认知问题。
- 用宏大归因替代具体修复。
- 忽略低层级的信息、工具、流程、权限和资源缺陷。
- 把弱势方无法改变的处境说成其“不愿承接”。

## 失效信号

事实未查清、单点错误、资源剥夺、制度不公、突发灾害或基本保障缺失时，不做宏大归因、不责备弱者；切换到事实核查、风险扫描或救急判断。

## 最高风险边界

不能把资源不足、规则压迫、基本保障缺失或单点故障归因为个人价值选择。

## 常组合

- `问题分型`：先判断问题类型。
- `主要矛盾`：多层问题并存时找主攻层。
- `会议签字固化承诺`：规则层需要明确承诺时。
- `规律代价对齐检查`：价值层涉及目标代价错位时。
