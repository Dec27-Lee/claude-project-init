# 前置决策校验

> 动作类型：执行验证
> 场景标签：产品、设计、AI、文档、研发、工作

## 适用处境

准备让人或 AI 直接生成原型、页面、文档、演示稿、代码结构、流程方案等最终产物，但关键边界、架构、骨架、验收口径还没有显性化；方向一旦错了会返工、污染结构或误导后续执行。

## 不适用处境

- 低风险一次性小改，已有明确模板和验收口径。
- 用户只要探索 Demo、方向样例或灵感草图，且明确接受不完整。
- 当前缺口是事实不足，需要先查证，而不是做前置决策。
- 前置决策无法被任何人检查，只会变成形式主义文档。

## 输入

- 最终产物类型和使用场景。
- 方向错误的返工成本、影响范围和不可逆程度。
- 必须先确定的边界：目标、对象、模块、信息层级、范式、权限、质量标准等。
- 谁能确认、确认到什么粒度、哪些问题必须停问。

## 操作步骤

1. 判生成成本：如果最终产物一旦生成会带来大返工、多人依赖或结构污染，则需要前置校验。
2. 列前置决策：目标/受众、主对象、结构边界、信息层级、技术/设计范式、验收口径、不可做项。
3. 设 checkpoint：哪些决策必须先被确认，哪些可由执行者自主，哪些命中就停问。
4. 先产出可审核的中间物：一句话目标、架构草图、模块定位、骨架、验收标准，而不是直接做最终物。
5. 放行后再生成：生成过程中不得偷偷改变已确认的前置结构；需要改变时回到 checkpoint。
6. 记录未确认假设：不能等同于已确认事实，后续验收时优先复查。

## 关键自问

- 现在直接生成，最可能把哪个错误固化？
- 哪些判断一旦错了，后面再改成本很高？
- 这个前置产物能被用户或团队快速审核吗？
- 我是在降低返工，还是给简单任务增加仪式？
- 哪些问题必须问，哪些可以先用明确假设推进？

## 输出格式

```markdown
## 前置决策
- 最终产物：
- 生成成本：低 / 中 / 高
- 必须先确认：
  1.
  2.
  3.
- 可自主假设：
- 必须停问：

## checkpoint
- 中间产物：
- 放行条件：
- 变更回退条件：

## 下一步
- 先做：
- 暂不生成：
```

## 常见误用

- 把前置决策写成冗长方案，而不是可审核的中间物。
- 对低风险小改也设置复杂 checkpoint。
- 前置决策没有通过条件，仍然无法判断能不能继续。
- 生成过程中悄悄改变已经确认的架构、骨架或边界。
- 把“用户没确认”当作不行动借口，而不是列明可自主假设。

## 失效信号

前置清单超过当前任务需要；用户已给出明确模板和验收；继续校验不会降低返工；事实缺口会改变方向但尚未查证；任务需要先产出样例激发讨论。命中时切换到 `最小行动`、`信息缺口定位` 或直接执行低风险步骤。

## 常组合

- `目标澄清`：前置目标和验收口径不清时。
- `模块一句话定位校验`：产物涉及模块、目录、页面组或职责边界时。
- `信息一二三档`：产物涉及页面/文档信息层级时。
- `质量裁决`：生成后判断是否允许进入下一阶段。
- `完成检查`：最终声明完成前检查是否覆盖原始需求。
