# 组件权力

> 动作类型：结构治理
> 场景标签：AI、组织、产品、协作、工作

## 适用处境

卡片、Feed、列表、待办、摘要、看板、提醒等组件选择会影响责任、解释权、控制感或组织压力，不只是视觉呈现。

## 输入

- 候选组件、对象性质、用户任务、状态和责任关系。
- 谁看、谁操作、谁被提醒、谁被评价。
- 组件是否涉及排序、摘要、公开状态、催办或自动决策。

## 操作步骤

1. 识别候选组件和它承载的主任务。
2. 判断组件把主动权交给谁：用户、系统、管理者、协作者或算法。
3. 判断组件新增谁的成本：解释、处理、确认、被评价、被催促。
4. 判断组件是否改变责任：谁被展示、谁需要证明、谁失去上下文。
5. 设计补救：可关闭、可解释、可申诉、可切换视图、降低强制性。
6. 输出采用、调整或放弃建议。

## 关键自问

- 这个组件让谁更轻松，让谁更紧张？
- 它是在帮助用户完成任务，还是替组织施压？
- 它是否制造了“看见即负责”的关系？
- 它是否把复杂判断伪装成简单排序？
- 用户有没有恢复控制的路径？

## 输出格式

```markdown
- 候选组件：
- 组件带来的权力变化：
  - 获益方：
  - 承担成本方：
  - 失去控制方：
- 主要风险：
- 补救设计：
- 是否建议采用：
```

## 常见误用

- 只从视觉和效率判断组件，不看权责后果。
- 把 Feed、待办、红点、排序当成中性工具。
- 用摘要替代原文，导致解释权被系统接管。
- 用看板公开状态，意外制造组织压力。

## 失效信号

组件只是静态展示，不影响操作、责任或解释权；用户和关系链路非常简单；问题核心是数据缺失，不是组件选择。

## 最高风险边界

不能用组件设计制造隐性强迫、羞辱、监控或绩效压力；不能让系统排序、摘要、待办替用户承担不可解释的判断责任；涉及人、绩效、关系、承诺、隐私的组件必须保留解释、纠错和退出路径。

## 常组合

- `责任状态`：处理状态展示带来的责任外溢。
- `默认值制度`：审计默认排序、默认提醒和默认展示。
- `页面范式选择`：判断组件是否适配主任务。
- `组织叙事校准`：防止组件服务口号而非任务。
