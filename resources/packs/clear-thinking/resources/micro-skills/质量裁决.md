# 质量裁决

> 动作类型：执行验证
> 场景标签：产品、研发、设计、选型、工作

## 适用处境

交付前需要判断 PRD、原型、代码方案、设计稿、技术选型或协作产物是否允许进入下一阶段；重点不是挑毛病，而是裁决通过、带风险通过还是阻塞。

## 输入

- 裁决对象、目标、下一环节和验收标准。
- 需求、架构、范式、字段、权限、状态、交互、异常和维护成本。
- 已知风险、限制和必须保留的取舍。

## 操作步骤

1. 明确裁决对象和下一阶段：上线、评审、开发、验收、移交或复盘。
2. 建立裁决维度：目标一致性、场景清晰度、流程闭环、字段/权限/状态完整性、异常边界、验收标准、维护成本。
3. 对每项判定 PASS / WARN / FAIL。
4. 区分阻塞项和警告项：FAIL 必须处理，WARN 可带风险通过但需后手。
5. 输出最小修复动作和复查标准。
6. 如果没有具体方案可检查，停止裁决，转入需求澄清或方案设计。

## 关键自问

- 这个交付现在要服务哪个下一环节？
- 哪些问题会导致下一环节无法开始？
- 哪些只是质量瑕疵，不应阻塞？
- 如果现在放行，谁承担风险？
- 验收标准是否能被第三方独立判断？
- 是否存在看起来完整但关键状态/异常缺失的问题？

## 输出格式

```markdown
## 裁决结论
- 总体：PASS / WARN / FAIL
- 适用下一阶段：

## FAIL 阻塞项
1. 问题：
   - 影响：
   - 最小修复：

## WARN 警告项
1. 风险：
   - 可接受条件：
   - 后手：

## PASS 项
- 已满足：

## 复查标准
-
```

## 常见误用

- 把质量裁决变成审美评价或主观偏好。
- 把所有瑕疵都判成 FAIL，导致交付无法推进。
- 没有区分阻塞项和警告项。
- 在没有具体方案时强行裁决。
- 只检查表面格式，不检查目标、流程、状态和验收。

## 失效信号

还没有具体方案可检查；裁决结果不能改变是否进入下一阶段；判断标准来自个人偏好而非目标和风险；PASS/WARN/FAIL 没有对应行动。

## 最高风险边界

不要把质量裁决当作权力压制工具；裁决必须绑定目标、风险和下一环节，高风险场景必须说明证据、影响和最小修复，而不是只给结论。

## 常组合

- `前置决策校验`：生成前先确认关键边界、骨架和放行条件，避免把方向错误带到质量裁决阶段。
- `需求四问`：先确认需求表达，再裁决方案。
- `字段-状态-操作卡`：检查产品/系统完整性。
- `完成检查`：交付前最后关口。
- `问题分型`：判断是质量问题、目标问题还是协作问题。
- `不可能三角识别`：质量、速度、范围冲突时裁决取舍。
