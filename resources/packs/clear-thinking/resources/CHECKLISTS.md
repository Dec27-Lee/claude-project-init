# clear-thinking 检查清单

本文件用于防止输出再次变浅、变教条：只讲原则、没有当前事实、没有主要矛盾、没有行动、没有验收，或者按关键词套方法。

## 1. 输出合格门槛

触发 clear-thinking 后，输出至少满足：

```markdown
- [ ] 是否判断了任务类型：简单执行 / 复杂判断 / 跨轮记录 / 大任务调度？
- [ ] 是否区分已知事实、合理推断、待确认假设？
- [ ] 是否指出当前主要矛盾，而不是平均罗列？
- [ ] 是否说明本轮采用哪些微技能、暂不采用哪些？
- [ ] 采用的微技能是否已读取对应文件，并核对“操作”和“失效信号”？
- [ ] 是否给出最小行动？
- [ ] 是否给出验收方式？
- [ ] 是否给出失败信号或止损条件？
- [ ] 是否判断是否需要 work-journal / Workflow / 子代理 / 专用技能？
```

## 2. 反教条检查

教条主义在本技能中指：**模型先于事实，答案先于问题，原则先于情境，表达先于验证**。

输出前检查：

```markdown
- [ ] 我是否先判断问题处境，而不是按关键词套方法？
- [ ] 我是否把当前事实放在框架之前？
- [ ] 我是否找到本轮主要矛盾，而不是展示很多方法？
- [ ] 我是否明确本轮不用哪些看似相关的方法？
- [ ] 我是否没有用书名、人物、公司案例、材料观点当权威背书？
- [ ] 我是否给出了可验证的最小行动？
- [ ] 如果现实反馈相反，我是否给出修正入口？
```

必须重写的失败模式：

- 输出像“分类器命中结果”。
- 只说“结合实际、具体分析、注意风险”，但没有当前事实和差异。
- 大量复述资料、人物、公司案例，却没有当前问题判断。
- 只新增方法名，没有操作步骤、适用边界和失效信号。
- 为了显得深刻调用过多微技能。
- 没有说明暂不处理项和不选方案。
- 没有最小行动、验收方式或失败信号。

## 3. 证据质量检查

| 等级 | 类型 | 输出方式 |
| --- | --- | --- |
| A | 用户明确提供的事实、文件、数据、运行结果 | 可作为事实依据 |
| B | 工作区规则、索引、命中历史记录 | 可作为流程和上下文依据 |
| C | 公开资料、可靠调研结果 | 可作为外部参考，必要时列 URL |
| D | 基于经验和模式的合理推断 | 必须标为“推断” |
| E | 未验证猜测、情绪判断、直觉 | 必须标为“待确认假设” |

自检：

```markdown
- [ ] 关键判断是否有 A/B/C/D/E 依据？
- [ ] D/E 是否被误写成确定事实？
- [ ] 是否需要补充资料？如果需要，是否先从索引定位？
- [ ] 是否有判断错的可观察信号？
```

## 4. 真实问题检查卡

```markdown
- [ ] 是否识别表层请求？
- [ ] 是否还原真实目标？
- [ ] 是否识别直接使用者、决策者、受影响者？
- [ ] 是否写清触发场景？
- [ ] 是否说明不解决的代价？
- [ ] 是否列出硬约束？
- [ ] 是否列出可调整变量？
- [ ] 是否标记未确认假设？
```

## 5. 主要矛盾检查卡

```markdown
- [ ] 是否判断当前阶段？
- [ ] 是否列出显性问题？
- [ ] 是否指出本阶段主要矛盾？
- [ ] 是否指出矛盾主要方面？
- [ ] 是否说明为什么它比其他问题更关键？
- [ ] 是否列出暂不处理项？
- [ ] 是否说明判断错了会出现什么信号？
```

## 6. 最小行动检查卡

```markdown
- [ ] 是否有一个最小行动，而不是一堆建议？
- [ ] 是否说明输入？
- [ ] 是否说明输出？
- [ ] 是否说明验收标准？
- [ ] 是否说明失败信号？
- [ ] 是否有回滚/后手？
- [ ] 是否说明下一轮根据什么调整？
- [ ] 如果是目标与代价问题，是否拆清现实条件、必要代价和主体承接能力？
- [ ] 如果是危机场景，是否拆清事实线、现金线、组织线、治理线，并先处理立即止血或专业支持？
- [ ] 如果问题反复修不好，是否区分了技术层、规则层、价值取舍层，并避免未查事实就宏大归因？
- [ ] 如果需要固化多人承诺，是否保证信息充分、可拒绝、非胁迫，并没有用记录/签字压服弱势方或替代必要法律程序？
```

## 7. 产品研发检查卡

产品/研发/页面/PRD/原型任务检查：

```markdown
- [ ] 是否回答需求四问：什么人、什么场景、什么问题、什么目的？
- [ ] 是否先判断复用、扩展还是新建？
- [ ] 每个模块是否有一句话定位？
- [ ] 是否列出一档/二档/三档信息？
- [ ] 是否选择页面/功能范式并说明依据？
- [ ] 字段是否有依据、类型、校验、默认值、权限、异常？
- [ ] 状态是否有触发条件、可执行操作、下一状态、回滚？
- [ ] 组件是否匹配数据形态和标准组件库？
- [ ] 关键操作是否有成功/失败反馈？
- [ ] 体验判断是否有真实高质量样本、可观察指标和市场/方案对照？
- [ ] 临摹标杆时是否拆清核心机制、可替代部分、知识产权和合规边界？
- [ ] 是否有 PASS / WARN / FAIL 质量裁决？
```

## 8. AI / 组织产品检查卡

涉及 AI 工作台、Agent、办公协同、平台产品、消息/待办、智能摘要、默认排序时检查：

```markdown
- [ ] 是否拆出用户发心、产品发心、组织发心、商业发心、技术发心？
- [ ] 本阶段主发心是否明确，哪些发心本轮被降级？
- [ ] 定位是否回答：为谁、在哪用、替代什么、为什么是我？
- [ ] 是否识别不可兼得目标，而不是把冲突下放给设计和研发？
- [ ] 组件是否改变了用户责任、心理成本、解释权或控制感？
- [ ] 已读、已处理、已触达等状态是否区分系统感知、用户确认和对外展示？
- [ ] 默认值是否可解释、可调整、可退出？
- [ ] 共创用户和真实用户是否区分，反馈是否用行为/投入验证？
- [ ] 高频迭代是在学习和校准，还是在制造敏捷债务？
```

## 9. 创业经营与长期判断检查卡

涉及创业、经营、增长、长期产品、职业选择或人生长期判断时检查：

```markdown
- [ ] 是否判断外部趋势是真变化、公开信号，还是热点想象？
- [ ] 用户是否被当作长期关系，而不是流量、指标或抽象画像？
- [ ] 是否只选 1-2 个主攻场景，并定义尖叫点与最低可交付边界？
- [ ] 效率提升是否减少系统损耗，而不是把压力转嫁给人？
- [ ] 口碑信号是否来自复用、推荐、留存、续费或异常场景中的信任？
- [ ] 当前动作是否积累能力、信任、资产、数据、系统、品牌或关系复利？
- [ ] 高收益路径是否拆清收益来源、反噬对象、退出窗口和止盈线？
- [ ] 新产品、新品牌、新团队或个人可信度是否有可查证据链，而不是虚假包装或空喊优势？
- [ ] 弱资源竞争是否找到了真实需求、主流强者不经济的缝隙和可承受失败成本，而不是低价内卷或透支人？
- [ ] 带有公益、情怀、帮扶或使命叙事的项目，去掉道德压力后是否仍有产品/服务价值，并满足透明披露要求？
- [ ] 多人多钱项目是否拆清出资、控制、亏损承担、实际执行、顾问角色、退出条件和知情同意，并排除规避监管、逃避债务、利益输送、挂名背锅？
- [ ] 长期扶持、赋能或生态合作是否拆清外部提供什么、主体承接什么、反馈机制和不可转嫁的基本责任？
- [ ] 是否只做决策结构辅助，而不是法律/医疗/心理/财务专业建议？
- [ ] 是否先设计低成本试探，而不是假装给终身答案？
```

## 10. 高上下文降噪检查卡

```markdown
- [ ] 痛感是否已分流为结构信号、个人状态、身体疲惫、关系问题或行动问题？
- [ ] 是否判断继续解释是在服务行动，还是延迟切断？
- [ ] 是否写清哪些责任不是本轮要承担的？
- [ ] 模糊帮忙请求是否拆清具体事项、边界、成本和不可承接责任？
- [ ] 外部帮助是否区分了可提供资源与主体必须自承接动作？
- [ ] 是否避免人格诊断、隐私推断和痛苦浪漫化？
```

## 11. 与工作区机制配合检查

```markdown
- [ ] 是否涉及记录、继续、上次、之前、进度、复盘、完成检查？如果是，是否使用 work-journal？
- [ ] 是否需要历史记录？如果是，是否先读 `local/work-journal/index.md`，并只读取索引命中的 `local/work-journal/records/*.md`？
- [ ] 是否涉及大任务、多材料、多阶段、交叉验证？如果是，是否考虑 Workflow / 子代理？
- [ ] 是否涉及新增长期资料入口？如果是，是否更新 `.claude/workspace-index.md`？
- [ ] 是否涉及专用技能领域？如果是，clear-thinking 是否只做前置判断？
```

## 12. 反浅层失败模式

出现以下情况要重写：

- “建议先明确目标，再分阶段推进”但没有当前目标和阶段判断。
- “要结合业务场景综合考虑”但没有角色、场景、任务、流程。
- “注意风险”但没有失败信号、止损条件和后手。
- 输出很多框架，却没有当前主要矛盾。
- 用书籍、人物、公司案例当权威背书。
- 把用户没说的隐私、心理、家庭、财务、组织内幕当判断依据。
- 产品问题不拆字段、状态、权限、组件、验收。
- 增长问题没有指标、实验、成功/停止标准。
- 复盘只总结过程，不提偏差、原因、下一轮动作和沉淀。
