# work-journal

当前工作区的轻量工作日志技能：用于杂事、文档、临时任务和少量代码任务的需求记录、过程续接、完成检查。

## 不触发场景

纯 Git 操作不使用本技能：用户仅要求提交、推送、拉取、同步当前工作区或分支时，直接执行 Git 流程，不创建、不更新工作记录。

只有用户明确要求记录、复盘或沉淀提交过程时，才进入 work-journal。

## 使用原则

1. **遵守工作区规则**：使用和维护索引的规则只以根目录 `CLAUDE.md` 为准。
2. **先读索引**：先读 `.claude/workspace-index.md`，再读 `local/work-journal/index.md`。
3. **记录按需读取**：`local/work-journal/records/` 下的历史记录不能默认全量读取，只能根据 `local/work-journal/index.md` 命中的记录路径按需读取。
4. **默认单文件**：多数任务只更新一个 `local/work-journal/records/YYYY-MM-DD-短标题.md`。
5. **新需求默认新记录**：除非用户明确说继续历史事项，或索引显示强相关，否则不要把新需求追加到旧记录。
6. **尊重只读意图**：用户明确要求“只读检查/只给建议/不要修改文件/不要落盘”时，不创建或更新记录，只回复检查结论。
7. **完成前检查**：只有覆盖需求、产物明确、验证说明清楚，才能说“已完成”。
8. **不全量遍历**：索引能定位时，不为了熟悉情况扫描全仓或全量读历史记录。

## 执行流程

### 1. 定位记录

- 新需求：先读 `local/work-journal/index.md` 判断是否命中强相关记录；如果没有且任务需要跨轮追踪、复盘或落盘，再创建新记录并更新索引。
- 继续任务：先读 `local/work-journal/index.md`，根据标题、关键词、状态、备注定位对应记录，只读取命中的 `local/work-journal/records/<文件名>.md`。
- 复盘/检查：如果是当前任务，读取当前记录；如果是历史任务，先用索引定位，不读取无关记录。
- 只读请求：用户明确不希望修改文件或落盘时，只做只读检查，不创建、不更新 record，也不补写索引。

### 2. 记录最小字段

记录文件只保留这些字段：

- 需求来源：记录用户本轮原始需求；不主动读取或记录个人邮箱、账号等身份信息。只有用户明确要求署名时，才记录用户提供的名称。
- 原始需求：用户要什么。
- 范围：本轮做 / 不做 / 待确认。
- 计划：3-7 条行动。
- 进展：做了什么、产物路径、关键决策。
- 检查：需求覆盖、产物明确、验证情况、风险、结论。

### 3. 完成检查

声明完成前必须回答：

- 用户一开始的需求是否全部覆盖？
- 新增/修改/删除的文件是否明确？
- 能测试的是否已测试？不能测试是否说明？
- 是否还有待确认、风险或下一步？
- 是否按 `CLAUDE.md` 判断过是否需要更新 `.claude/workspace-index.md`？
- 如果读取了历史工作记录，是否是通过 `local/work-journal/index.md` 定位后按需读取？

结论只能是：`已完成` / `部分完成` / `未完成`。

## 与 clear-thinking 的配合规则

`work-journal` 负责记录、续接、检查和落盘；`clear-thinking` 只在需要判断、取舍、复盘归因时提供短结论，不替代记录主流程。

### 1. 调度顺序

涉及记录、继续、复盘、完成检查时：

1. 先由 `work-journal` 通过索引定位需求，并判断是否需要创建或更新记录。
2. 再判断是否需要 `clear-thinking`。
3. 再判断 `clear-thinking` 应在主会话、独立子代理还是 Workflow 中执行。
4. 如果本轮需要落盘，最后把短结论回写记录，并继续执行或给出完成判定；只读请求则只回复结论。

不要反过来先展开分析，再补做记录。

### 2. 何时不调用 clear-thinking

以下情况默认不调用：

- 单点明确小任务、一次性短答、低风险直接操作。
- 用户明确要求“别分析，直接做”。
- 记录内容只是机械更新路径、状态或简单进度。
- 当前任务已有专用技能必须优先执行，且不需要额外判断。

### 3. 主会话直接调用 clear-thinking

实现方式：在主会话中通过 `clear-thinking` 技能处理压缩后的 `thinking-brief`，不要把完整记录、长聊天或方法论全文塞入输入。

仅当满足以下条件时，才在主会话直接调用：

- 只依赖当前用户输入或当前 record 的少量内容。
- 不需要展开读取多个文件或多个历史记录。
- 输出只用于补充“范围、优先级、最小行动、完成判定”。
- 预计输出可控制在短结论内。

适用场景：

- 新需求目标不清，需要轻量判断真实目标和本轮范围。
- 复盘/检查只需要判断是否完成、缺口是什么。
- 大任务前只需要先判断是否应升级 Workflow。

### 4. 使用独立子代理调用 clear-thinking

满足任一条件时，优先用独立子代理处理，避免占用主会话上下文：

- 需要读取 2 份及以上材料或历史记录。
- 需要从历史记录中比对前后状态、目标变化或偏差原因。
- 需要读取 `.claude/skills/clear-thinking/resources/ROUTER.md`、`MICRO_SKILLS.md`、`CHECKLISTS.md` 或命中的 `micro-skills/<微技能>.md` 中任一资料文件并展开分析。
- 需要比较多个方案、多个判断口径或多个风险视角。
- 分析过程会明显膨胀主会话上下文，但最终只需要短结论写回记录。

实现方式：主会话启动独立子代理，把 `thinking-brief`、允许读取的最小文件路径和输出格式交给子代理；子代理可按需读取 `clear-thinking` 技能入口和 `.claude/skills/clear-thinking/resources/` 下的必要资料，但不得全量读取 `local/work-journal/records/`。

若当前环境无法使用子代理，则在主会话只读取最小必要材料，压缩为短结论；不要为了模拟子代理而展开全量资料。

子代理只返回摘要；主会话不保留长篇推理过程。

### 5. 何时升级为 Workflow

是否升级以本文末尾“何时升级为 Workflow”为准。若升级，先按索引定位并在需要时记录需求，再把 `clear-thinking` 作为 Workflow 中的分析节点，而不是在主会话里长篇展开。

### 6. 传给 clear-thinking 的最小信息

默认只传 `thinking-brief`，不要传全量上下文：

```markdown
## thinking-brief

- 场景类型：新需求 / 继续历史 / 复盘检查 / Workflow 前置 / 简单任务
- 用户原始请求：
- 当前记录路径：
- 命中历史记录：无 / `local/work-journal/records/xxx.md`
- 已知事实：最多 5 条
- 当前目标：1 条
- 硬约束：最多 5 条
- 待确认假设：最多 3 条
- 需要 clear-thinking 判断的问题：
  1. 真实目标是什么？
  2. 主要矛盾是什么？
  3. 最小行动是什么？
  4. 验收方式是什么？
  5. 是否需要 Workflow / 子代理 / 专业技能？
- 输出限制：
  - 不全量读取 `local/work-journal/records/`
  - 不输出长方法论
  - 返回可写入工作记录的 5-8 条摘要
```

不要默认传递：

- 全量聊天记录。
- `local/work-journal/index.md` 的无关内容。
- 无关文件、无关背景材料。
- 已经完成的长篇中间思考。

### 7. clear-thinking 返回格式

默认要求 `clear-thinking` 返回可直接写回记录的短格式：

```markdown
## clear-thinking 摘要

- 真实目标：
- 主要矛盾：
- 本轮最小行动：
- 验收方式：
- 风险/失败信号：
- 是否需要 Workflow：
- 是否需要子代理：
- 是否需要更新索引/沉淀资产：
```

如用于完成检查，可追加：

```markdown
## 完成判定

- 结论：已完成 / 部分完成 / 未完成
- 主要缺口：
```

### 8. 回写 journal 的规则

将 `clear-thinking` 的结果回写到 record 时：

- 只记录结论、关键依据、行动和风险。
- 不整段复制分析过程。
- 不把方法论展开文本写入记录文件。
- 保持记录可续接、可检查、低噪声。

## 可选自动提醒资源

本 pack 会复制一个轻量 hook 示例脚本，但默认不修改 `.claude/settings.json`，也不自动启用 hook。

- `UserPromptSubmit`：遇到需求/记录/继续/检查/复盘类提示时，可向 Claude 可见上下文注入提醒。
- `PreCompact`：上下文压缩前可提醒补记录。

脚本位置：`.claude/skills/work-journal/resources/hooks/reminder.py`。

hook 只提醒，不自动写入用户内容。若要启用，请先人工审查脚本，再手动合并 settings，避免误记、噪声和隐私问题。

## 何时升级为 Workflow

满足以下任一条件时，优先在 work-journal 定位并按需记录需求后，使用 Claude Workflow / Dynamic workflows 做流程编排；Workflow 优先级高于计划模式：

- 跨多个子项目/仓库/阶段。
- 涉及多个业务线、多份材料或多类输入源。
- 需要调研、设计、实现、验证等多轮推进。
- 需要“定位记录 → 分析 → 执行 → 验证 → 回写”的完整编排。
- 单个对话难以稳定协调，或需要多个子代理交叉检查。
- 用户明确要求完整审计、大型迁移、长期项目工作台或使用 workflow。

调度边界：

| 场景 | 处理方式 |
| --- | --- |
| 低风险小任务 | 主会话直接执行，必要时只写一条记录 |
| 目标不清但材料少 | 主会话用 `clear-thinking` 输出短结论 |
| 多材料、多历史记录或多判断口径 | 子代理读取最小材料并返回摘要 |
| 多阶段、多产物、多验证闭环 | Workflow 编排，`clear-thinking` 和子代理作为节点 |

不满足上述条件的中小任务，可以在 work-journal 之后进入计划模式；低风险小任务可直接执行。若当前环境无法使用 Workflow，则退回主会话计划 + 必要子代理，但必须控制上下文读取范围，并在记录中说明未使用 Workflow 的原因。
