# thinking-distiller 与 clear-thinking 接入规则

本文件规定蒸馏出的微技能如何安全接入 `.claude/skills/clear-thinking/resources`。

核心原则：**先查重，后新增；先合并，后扩展；先测试，后写入；一微技能一文件；按动作类型和标签路由，不按单一场景固定归类。**

---

## 1. clear-thinking 2.7.0 当前架构

`clear-thinking` 的运行链路是：

```text
SKILL.md → ROUTER.md → MICRO_SKILLS.md → micro-skills/<微技能>.md → 输出判断/行动
```

其中：

| 文件 | 责任 |
| --- | --- |
| `SKILL.md` | 技能入口、执行协议、输出门槛 |
| `resources/ROUTER.md` | 按问题形态选择判断动作类型和候选组合 |
| `resources/MICRO_SKILLS.md` | 微技能级索引，列出动作类型、触发信号、产出、场景标签、常组合和具体文件路径 |
| `resources/micro-skills/<微技能>.md` | 实际可调用微技能内容，一微技能一文件 |
| `resources/CHECKLISTS.md` | 完成前检查和反教条检查 |

蒸馏时默认新增或修改具体微技能文件，并同步更新 `MICRO_SKILLS.md`。只有新增判断动作类型、组合路径或停止条件时，才修改 `ROUTER.md`。

---

## 2. 动作类型映射

微技能不按工作、生活、产品、经营等使用场景固定归类。场景只作为标签，帮助检索和排序；真正的接入轴是**这个微技能解决哪类判断缺口、产出什么中间结论**。

| 蒸馏出的方法类型 | 推荐动作类型 | 示例动作 |
| --- | --- | --- |
| 事实、证据、样本、信息缺口 | 事实校准 | 失败路径反推、噪声事实分层、外行解释校准 |
| 目标、优先级、代价、不可兼得 | 目标取舍 | 目标重写、代价对齐、选择标准制定 |
| 问题类型、主要矛盾、层级诊断 | 问题诊断 | 层级诊断、主矛盾定位、根因分层 |
| 风险、边界、止损、危机 | 风险边界 | 责任边界、止损线、危机拆线 |
| 权责、机制、治理、承接关系 | 结构治理 | 控制权校准、承接链设计、默认值责任审查 |
| 需求、页面、字段、体验、表达 | 产品表达 | 场景反推字段、异常状态优先、样本拆核 |
| 用户、口碑、可信度、长期资产 | 增长信任 | 信任证据链、口碑飞轮、复利路径判断 |
| 行动、验收、质量、复盘 | 执行验证 | 最小行动、质量裁决、完成检查 |
| 关系、人情、帮助、主体承接 | 关系承接 | 朋友请求边界、责任承接、解释关闭 |
| 从材料中抽象方法 | 方法建设 | 判断动作提取、微技能封装、反权威背书 |

如果一个方法横跨多个使用场景，保留一个微技能，用多个 `scene_tags` 标注；如果它横跨不同判断动作类型并产生不同产出，应拆成多个微技能。

## 3. 写入规则

### 3.1 创建或修改独立微技能文件

每个运行层微技能单独保存为：

```text
.claude/skills/clear-thinking/resources/micro-skills/<微技能名>.md
```

写入格式必须与现有 2.7.0 微技能一致；新增或修改运行层微技能默认使用可执行微技能卡，而不是只写一句操作或短骨架：

```markdown
---
name: 微技能名
action_type: 动作类型
scene_tags: [工作, 生活, 产品]
output: 该技能产生的中间结论
---

# 微技能名

> 动作类型：动作类型
> 场景标签：工作、生活、产品

## 适用处境
## 不适用处境
## 输入
## 操作步骤
## 关键自问
## 输出格式
## 常见误用
## 失效信号
## 最高风险边界（高风险/易误用时必填）
## 常组合
```

要求：

- 微技能名使用动作名，不默认使用人物名或书名。
- `action_type` 表示判断动作类型，不写成单一使用场景。
- `scene_tags` 可以同时包含工作、生活、产品、经营、组织、关系等多个场景，不作为调用边界。
- `操作步骤` 写成 3-6 个判断动作，避免只剩口号或一句提示词。
- `关键自问` 用于稳定 AI 的执行过程，避免只引用微技能名。
- `输出格式` 要能产出中间结论，便于后续组合。
- `失效信号` 必须具体，不能写“任何时候都适用”。
- 高风险或易误用微技能必须额外写清最高风险边界和升级/停止条件；低风险微技能也要保留清晰失效信号。
- 不写来源、引文、人物故事和权威背书。
- 不把多个可独立触发、不同产出的微技能写入同一个文件。
- 不因“这是产品/经营/生活材料”而寻找固定场景目录；目录已扁平化，场景只作为标签。

### 3.2 更新 MICRO_SKILLS.md

当新增或移动微技能后，在 `MICRO_SKILLS.md` 的微技能路径索引中加入或更新一行：

```markdown
| 动作类型 | 微技能 | 触发问题形态 | 产出 | 场景标签 | 常组合 | 文件 |
```

要求：

- 路径必须指向具体文件，而不是目录或聚合文件。
- 触发信号只写足以路由的短描述，不写完整方法。
- `产出` 写该微技能带来的中间结论，便于组合时判断是否需要它。
- `场景标签` 只用于检索，不限制调用范围。
- 如果只修改已有微技能内容且路径、动作类型、标签不变，可以不改索引。

### 3.3 更新 ROUTER.md

只有出现以下情况才修改：

- 新增了判断动作类型。
- 新增了典型组合路径。
- 新微技能改变了触发边界、停止条件或跨类型组合方式。
- 原有路由原则会误导为按场景固定目录调用。

不要因为新增 1 个普通微技能就修改路由器；但若该微技能应成为常用组合节点，需要同步更新。

### 3.4 来源、草案和过程文件放哪里

运行层不保留来源噪声。正式蒸馏任务默认把过程文件放在：

```text
local/thinking-distiller/sessions/YYYY-MM-DD-短标题/
```

推荐分工：

- `brief.md`：任务简报、材料范围、目标动作类型和场景标签。
- `material-map.md`：材料地图和证据强度。
- `extraction-cards.md`：判断动作提取卡。
- `candidate-skills.md`：候选微技能草案，可保留来源线索、来源主体、原文精度、误用风险和接入状态。
- `precision-review.md`：原文精度、主体归因、命名、拆分/合并和暂缓接入复核。
- `candidate-skills-v2.md`：精度复核后有重大修正时使用。
- `integration-plan.md`：写入 clear-thinking 前的补丁计划。
- `tests.md`：正例、反例和切换场景测试。

索引登记在：

```text
local/thinking-distiller/index.md
```

工作日志只写需求、进展、路径和完成检查，不复制完整蒸馏过程。不把大段来源材料放入 `.claude/skills/clear-thinking/resources`。

## 4. 查重规则

写入前必须回答：

```markdown
- 目标微技能和已有哪个微技能最像？
- 是否已读取这些相近微技能的具体文件，而不是只看 `MICRO_SKILLS.md`？
- 它是补充现有动作，还是产生不同中间结论？
- 它的主动作类型是什么？是否需要多个场景标签？
- 如果合并，会不会让已有微技能变臃肿？
- 如果新增，会不会造成路由混乱？
- 如果新增为独立文件，MICRO_SKILLS.md 是否能按触发信号和产出精确路由到它？
```

处理优先级：

1. 放弃：只是概念或口号。
2. 合并：只是已有微技能的一条操作补充。
3. 新增：触发处境、操作、产出和失效信号都独立。
4. 新增动作类型：只有现有动作类型完全无法承载时才考虑。

### 4.1 拆分/合并规则

必须拆分的情况：

- 同一个候选有两个以上不同产出。
- 同一个候选跨越两个触发处境，且需要不同操作或失效信号。
- 同一个候选需要不同失效信号。
- 同一个候选在不同判断动作类型中承担不同作用。
- 同一个文件内会出现两个可独立调用的方法。

应该合并的情况：

- 只是同一判断动作的一个步骤。
- 触发处境、输入、产出、失效信号基本一致。
- 新增后只会制造路由混乱，没有独立中间结论。

### 4.2 接入状态

每个候选在补丁计划中必须标注状态：

| 状态 | 含义 |
| --- | --- |
| `accept` | 可接入运行层 |
| `merge` | 并入已有微技能 |
| `split` | 先拆分再评估 |
| `revise` | 需改名、收窄或补证据 |
| `defer` | 暂缓接入，保留候选 |
| `reject` | 放弃，不进入运行层 |

`defer` 触发条件：原文场景过窄但有潜在价值；误用后成本高；需要更多反例测试；涉及高上下文伦理/责任边界；路由器容易误触发。

`defer` 必须记录：暂缓原因、主要误用方式、最小补充材料、复审条件、是否允许留在候选索引。

## 5. 推荐接入流程

```text
候选微技能 → 原文支撑评分 → 抽象层级检查 → 命名/主体归因检查 → 查 MICRO_SKILLS → 读相近微技能文件 → 判定动作类型/场景标签/产出 → 查重/拆分/合并 → 反例与切换测试 → 接入状态判定 → 写入独立文件 → 更新索引 → 回写记录
```

### 最小工具读取

- 不全量读取 `micro-skills/`。
- 先读 `ROUTER.md` 和 `MICRO_SKILLS.md`。
- 再只读命中的相近微技能文件；采用、合并、扩展前必须核对具体文件中的操作步骤和失效信号。
- 如果需要 8 个以上微技能，或需要多材料、多阶段、交叉验证，优先用子代理或 Workflow。

---

## 6. 测试接入效果

每个准备接入的新增微技能至少用正例、反例和切换问题测试；暂缓候选必须测试误用场景和复审条件：

```markdown
## 测试问题

- 测试类型：正例 / 反例 / 切换 / 暂缓复审
- 问题：
- clear-thinking 应该先判断的动作类型：
- 场景标签：
- 应调用的微技能组合：
- 新微技能的作用：
- 输出应产生的中间结论：
- 什么时候停止：
- 不用它时应切换到什么：
```

合格标准：

- 它能被 `MICRO_SKILLS.md` 精确定位到具体文件。
- 它能被 `ROUTER.md` 放入合理动作类型和候选组合。
- 它能和已有微技能组合，而不是替代全流程。
- 它能产生明确中间结论。
- 它有明确失效信号。
- 不需要引用来源也能成立。

---

## 7. 完成检查

写入完成后检查：

- [ ] 用户原始要求是否覆盖？
- [ ] 新增/修改文件路径是否明确？
- [ ] 是否只读取了必要的 clear-thinking 文件？
- [ ] 是否在采用、合并或修改前读取了相近的具体微技能文件，并核对操作步骤、输出格式、常见误用和失效信号？
- [ ] 是否更新了 `MICRO_SKILLS.md`？如果没更新，原因是什么？
- [ ] 是否需要更新 `ROUTER.md`？如果没更新，原因是什么？
- [ ] `MICRO_SKILLS.md` 中的路径是否都指向存在的具体微技能文件？
- [ ] 是否需要更新 `.claude/workspace-index.md`？
- [ ] 是否已说明来源材料没有作为权威直接进入运行层？
- [ ] 是否有原文支撑评分、来源主体归因和命名审查？
- [ ] 是否判断过动作类型、场景标签、拆分/合并/暂缓接入状态？
- [ ] 是否有正例、反例、切换测试和失败信号？
