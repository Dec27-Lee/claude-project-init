# thinking-distiller 蒸馏协议

本协议用于把书籍、文章、演讲、访谈、课程、人物资料中的可迁移思考方法，蒸馏为 `clear-thinking` 可调用的微技能。

核心原则：**提取判断动作，不摘录思想概念；封装微技能，不复述来源权威。**

---

## 0. 过程产物落盘规则

正式蒸馏任务的过程产物默认写入：

```text
local/thinking-distiller/sessions/YYYY-MM-DD-短标题/
```

建议文件：

| 文件 | 用途 | 是否必需 |
| --- | --- | --- |
| `brief.md` | 蒸馏任务简报、范围、目标动作类型和场景标签 | 是 |
| `material-map.md` | 材料地图、章节/片段到问题场景的映射 | 是 |
| `extraction-cards.md` | 判断动作提取卡 | 是 |
| `candidate-skills.md` | 候选微技能草案，可保留来源线索 | 是 |
| `precision-review.md` | 原文精度、主体归因、命名、拆分/合并和误用风险复核 | 准备接入或用户要求校准时必需 |
| `candidate-skills-v2.md` | 精度复核后的候选版本 | 有重大修正时建议 |
| `integration-plan.md` | 接入 clear-thinking 的补丁计划 | 准备写入时必需 |
| `tests.md` | 工作/生活/反例场景测试 | 准备写入时必需 |

每个过程目录创建后，必须在 `local/thinking-distiller/index.md` 登记。不要把这些过程文件放进 `.claude/skills/thinking-distiller/`，也不要把来源噪声写入 `.claude/skills/clear-thinking/resources/`。

## 1. 阶段一：目标校准

### 输入

- 用户提供的材料：书名、文件、章节、摘录、笔记或资料目录。
- 用户希望服务的问题域：工作、生活、产品、经营、组织、长期判断、内耗、复盘等。
- 是否允许写入 `.claude/skills/clear-thinking/resources`。

### 操作

1. 确认用户要的是：摘要 / 读书笔记 / 方法蒸馏 / 微技能接入。
2. 如果是方法蒸馏，明确目标不是“学某人说话”，而是“提取可迁移判断动作”。
3. 把目标问题域拆成 `clear-thinking` 的动作类型和场景标签；场景标签只用于检索，不作为调用边界：
   - 事实校准：事实、证据、样本、信息缺口
   - 目标取舍：目标、代价、优先级、不可兼得
   - 问题诊断：问题类型、主要矛盾、层级诊断
   - 风险边界：风险、止损、危机、责任边界
   - 结构治理：权责、机制、治理、承接关系
   - 产品表达：需求、页面、字段、体验表达
   - 增长信任：用户、口碑、可信度、长期资产
   - 执行验证：行动、验收、质量、复盘
   - 关系承接：关系、人情、帮助、主体责任
   - 方法建设：从材料中抽象方法
4. 设定本轮产物：蒸馏草案 / 候选微技能 / clear-thinking 补丁 / 测试报告。

### 产出

```markdown
- 本轮材料：
- 目标问题域：
- 目标 clear-thinking 动作类型/场景标签：
- 本轮做：
- 本轮不做：
- 是否写入 clear-thinking：是/否/待确认
```

## 2. 阶段二：材料地图

### 输入

- 原始材料或可读取文件。
- 用户指定的优先章节或人物。

### 操作

按“问题场景”而不是“章节摘要”建立材料地图：

| 材料位置 | 表层主题 | 真实问题场景 | 可能包含的判断动作 | 证据强度 |
| --- | --- | --- | --- | --- |
| 第 X 章 / 文件名 | 讲什么 | 作者当时在解决什么问题 | 如何判断/取舍/验证 | 高/中/低 |

证据强度判断：

- **高**：材料中有具体情境、决策、行动、反馈。
- **中**：材料中有原则和案例，但反馈不完整。
- **低**：只有观点、口号、二手转述或语录。

### 产出

材料地图和候选提取点清单。低证据材料默认不进入微技能，只作为背景。

---

## 2.5 阶段二点五：原文精度锚定与主体标注

### 目标

在抽象成微技能之前，先约束候选动作的证据边界，避免把读者感受、人物气质或单句观点过度泛化为方法。

### 操作

每个候选动作进入提取卡前，先回答：

```markdown
- 具体片段：来自哪些材料位置？
- 真实问题：材料中的人物/作者当时在解决什么问题？
- 原文明确支持：原文中实际发生了什么行动、安排、取舍、反馈？
- 蒸馏者推断：哪些是根据上下文补出的解释？
- 来源主体：谁做出了判断动作？
- 辅助主体：谁提供了样本、资源、约束或执行建议？
- 主体边界：这是某个人的稳定方法，还是多人协作结果？
- 不可迁移部分：哪些依赖特殊身份、资源、时代或剧情条件？
```

### 准入

- 强支撑：有具体情境、决策、行动、约束和反馈，可进入候选。
- 中支撑：有场景和原则，但动作链需少量补全，可进入候选但必须标注推断。
- 弱支撑：只有观点、口号、台词、气质或二手转述，不直接进入候选。

---

## 3. 阶段三：判断动作提取

### 输入

- 材料地图中证据强度为高/中的片段。

### 操作

对每个片段回答：

```markdown
1. 作者/人物面对的真实问题是什么？
2. 来源主体是谁？辅助主体是谁？谁只是观察或评价？
3. 他先看到了什么事实或信号？
4. 他忽略了什么噪声或暂时不处理什么？
5. 他用什么判断动作改变了问题结构？
6. 这是风险识别、边界设置、结构设计、资源定位、验证机制、体验临摹，还是其他动作类型？
7. 如果是结构设计：参与方、权责关系、激励/约束、承接机制、反馈闭环分别是什么？
8. 他如何做取舍？
9. 他如何验证或等待反馈？
10. 这个动作如果迁移到用户工作/生活问题，触发条件是什么？
11. 哪些场景下这个动作会失效或误伤？
12. 这个候选是否可能需要拆分、合并、改名或暂缓？
```

### 不合格提取

以下内容不能直接变成微技能：

- “保持长期主义”“第一性原理”“逆向思维”这类概念名。
- “某人认为……”这类观点陈述。
- 无法说明触发条件、操作步骤、产出的格言。
- 依赖特定历史地位、资源禀赋或时代红利的做法。

### 合格提取

合格提取必须是动作：

```text
当信息噪声过多时，先把问题拆成“能改变决策的事实 / 只改变情绪的解释 / 需要验证的假设”，再决定是否继续收集信息。
```

## 4. 阶段四：动作归一与去来源化

### 目标

把“某本书/某个人的观点”降级为“可被 clear-thinking 调用的判断动作”。

### 操作

1. 去掉人物口吻和书名背书。
2. 把抽象名词改写成可执行动词。
3. 把价值判断改写成使用边界。
4. 把宏大原则切成 1-3 个可调用微动作。
5. 只保留必要的来源索引到建设记录，不写入运行层。

### 转换示例

| 来源化表达 | 运行层表达 |
| --- | --- |
| 芒格说要逆向思考 | 当正向求解陷入惯性时，先列出失败路径，再反推必须避免的关键条件 |
| 费曼学习法强调讲给别人听 | 当概念看似理解但无法应用时，用面向外行的解释暴露缺口 |
| 达利欧强调原则 | 当反复遇到同类冲突时，把一次性判断沉淀为可复查规则，并记录失效条件 |

## 4.5 阶段四点五：拆分/合并、命名审查与误用预演

### 拆分/合并判断

在封装成微技能前，必须判断候选是否过粗或过碎：

- 如果同一候选包含不同触发处境、不同产出或不同失效信号，应拆分。
- 如果只是已有微技能的一条操作补充，应合并。
- 如果一个泛名遮住了多个动作，应先拆分再命名。
- 如果误用成本高但边界尚不稳定，应标记为暂缓接入。

### 命名审查

候选名必须通过以下检查：

1. 能看出具体动作，而不是抽象价值词。
2. 保留原文最独特的判断机制，而不是只写“风险/原则/方法”。
3. 限定问题结构，不暗示适用于一切问题。
4. 不暗示错误来源主体。
5. 不诱导权谋化、道德化、宏大归因或不作为。

高风险命名包括：过泛名、过宏大名、只写风险名、只写结果名、人物背书名。

### 误用预演

每个候选草案至少写清：

```markdown
- 最危险误用：
- 最容易过度泛化的表达：
- 必须写进失效信号的边界：
- 哪类场景下宁可不用：
- 不用时应切换到哪些已有微技能：
```

## 5. 阶段五：微技能封装

### 标准格式

写入 `clear-thinking` 前，必须封装成适配 clear-thinking 2.7.0 的可执行微技能卡。新增或修改运行层微技能默认保持完整小节；低风险轻量技能可以压缩正文，但不能回退成一句操作或短骨架：

```markdown
---
name: 微技能名
action_type: 动作类型
scene_tags: [工作, 生活, 产品]
output: 该技能产生的中间结论
---

# 微技能名

> 动作类型：动作类型
> 场景标签：工作、生活、产品

## 适用处境

什么问题形态下调用；写“用户卡住的状态”，不要只写关键词。

## 不适用处境

哪些情况不要调用，尤其是高风险、专业边界、低风险直接执行或事实不足场景。

## 输入

需要哪些事实、材料、角色、约束和待确认假设。

## 操作步骤

1. 第一步判断什么。
2. 第二步拆什么变量。
3. 第三步形成什么中间结论。
4. 必要时说明何时切换到其他微技能。

## 关键自问

- 本技能要回答的关键判断问题是什么？
- 哪个条件不满足时判断会失效？
- 是否存在权力不对等、合规或安全边界？

## 输出格式

- 当前判断：
- 关键依据：
- 待确认项：
- 下一步：
- 失败/切换信号：

## 常见误用

不要把本技能用成什么。

## 失效信号

何时不用、停止、追问、查证、切换技能或交给专业支持。

## 最高风险边界

高风险或易误用微技能必填；低风险可省略本节或写“不适用”。

## 常组合

- 已有微技能：组合原因。
```

高风险微技能必须额外写“最高风险边界”，明确绝不能用本技能做什么。低风险轻量微技能可以适度压缩正文，但不能省略适用处境、输入、操作步骤、关键自问、输出格式、常见误用、失效信号和常组合。

### 命名规则

优先使用动作名，不使用人物名：

- 好：`失败路径反推`、`外行解释校准`、`重复冲突成规`、`噪声事实分层`
- 避免：`芒格逆向思维`、`费曼法`、`达利欧原则`

只有当同名动作容易混淆，才在建设记录中保留来源标签。

## 6. 阶段六：查重、合并与路由

### 查重顺序

1. 读 `clear-thinking/resources/MICRO_SKILLS.md` 看是否已有相近微技能、动作类型、场景标签、产出和具体文件路径。
2. `MICRO_SKILLS.md` 只能定位候选；采用、合并或修改某个微技能前，必须只读相近的具体微技能文件，并核对其操作步骤、输出格式、常见误用和失效信号。
3. 判断候选微技能是：
   - 合并到已有微技能；
   - 作为已有微技能的操作补充；
   - 新增为独立微技能；
   - 放弃，不写入运行层。

### 新增判断

只有满足以下至少一条才新增：

- 现有微技能无法表达该触发处境。
- 该动作产出的中间结论不同。
- 该动作有独特失效信号。
- 它能明显改善 clear-thinking 的组合能力。

---

## 7. 阶段七：场景测试

每个候选微技能至少测试 3 类场景；不能只给整组候选做代表性测试：

1. **正例场景**：至少 1 个工作或生活问题，能明确触发该技能。
2. **反例场景**：至少 1 个看似适用但实际不该用的场景。
3. **切换场景**：不用该技能时，应切换到哪些已有微技能。

高风险候选至少 2 个反例：一个检验“逃避/不作为”，一个检验“权谋化/道德化/宏大归因”等误用。

测试问题：

```markdown
- 能否明确触发？
- 是否能与 1-3 个已有微技能组合？
- 是否能产生中间结论，而不是直接给鸡汤？
- 是否知道何时停止？
- 是否去掉了来源崇拜？
```

## 8. 阶段八：接入和完成检查

### 写入前

- 用户明确允许修改 `clear-thinking` 运行资源。
- 已读取目标文件。
- 已准备补丁计划。
- 已完成原文精度锚定、主体归因、命名审查和误用预演。
- 已按索引定位并读取相近的具体微技能文件；没有只看索引就做合并/采用判断。
- 已通过质量检查，每个候选至少有正例、反例和切换测试。
- 已判定候选状态：接入 / 合并 / 拆分 / 修订 / 暂缓 / 放弃。

### 写入后

更新：

- 目标 `micro-skills/<微技能>.md` 独立文件。
- 如果新增、移动或重命名微技能，需要更新 `MICRO_SKILLS.md`。
- 如果新增分类或改变路由，需要更新 `ROUTER.md`。
- 如果新增长期入口，检查是否需要更新 `.claude/workspace-index.md`。
- 如果任务需要记录，更新 `local/work-journal/`。

### 完成声明

必须说明：

- 蒸馏自哪些材料。
- 新增/修改了哪些文件。
- 产生了哪些微技能。
- 如何被 `clear-thinking` 调用。
- 做过哪些测试。
- 哪些内容没有写入以及原因。
