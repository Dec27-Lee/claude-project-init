# thinking-distiller

面向 `clear-thinking` 的方法蒸馏技能：当用户提供书籍、文章、演讲、访谈、人物材料或研究资料，希望从中提取可复用思考方法，并沉淀为 `.claude/skills/clear-thinking/` 可调用的微技能时使用。

本技能的目标不是做读书摘要、名人语录、角色扮演或知识卡片，而是把材料中的**判断动作**提取成可路由、可组合、可验证、可停止的微技能。

```text
材料 → 原文精度锚定 → 判断动作 → 微技能草案 → 精度复核 → 路由归类 → 场景测试 → 写入 clear-thinking
```

核心原则：**先保真，再抽象；先标清谁在什么场景下做了什么设计，再去来源化为可迁移动作。**

当前适配目标是 `clear-thinking` 2.7.0：运行层已经是扁平的一微技能一文件结构，按“动作类型 + 触发信号 + 产出 + 场景标签 + 常组合”路由；`MICRO_SKILLS.md` 只能定位候选，采用或合并某个微技能前必须读取对应 `micro-skills/<微技能>.md`，并核对其操作步骤、输出格式、常见误用和失效信号。

## 0. 核心定位

- **负责蒸馏**：从书籍/资料中提取作者如何观察问题、识别主次、做取舍、设边界、验证行动。
- **负责封装**：把可迁移方法封装成 `clear-thinking` 的可执行微技能卡：适用处境、不适用处境、输入、操作步骤、关键自问、输出格式、误用边界、失效信号和常组合。
- **负责降噪**：运行层不保留书名、人名、语录和权威背书；来源只留在建设记录或溯源文件中。
- **负责接入**：按 `clear-thinking` 现有路由架构，把新微技能写成独立微技能文件，并在索引中标注动作类型、触发信号、产出、场景标签和常组合，必要时更新路由。
- **不负责替代 clear-thinking**：本技能只建设方法库，真正处理用户问题仍由 `clear-thinking` 路由和组合。
- **不负责全文摘录**：不批量复制受版权保护内容；只保留短引用、页码、章节位置和自己的结构化提炼。

## 1. 触发场景

优先用于：

- 用户提供一本或几本书，希望“提炼方法”“蒸馏成技能”“让 clear-thinking 可调用”。
- 用户提供人物资料，希望从中提取可迁移的判断方式，而不是模仿口吻。
- 用户要求把某套思想、方法论、课程、笔记转成微技能或检查清单。
- 用户指出已有 `clear-thinking` 方法不够，需要从新材料补充方法。

不要用于：

- 只要读书摘要、导读、金句、章节概括。
- 只要 AI 扮演某个人或模仿文风。
- 没有材料、也不需要沉淀技能的一次性判断。
- 用户明确只要建议，不要落盘或不要改技能文件。

## 2. 与 clear-thinking 的关系

`clear-thinking` 是运行时的“微技能操作系统”；`thinking-distiller` 是建设时的“方法蒸馏工厂”。

| 层级 | 责任 | 文件 |
| --- | --- | --- |
| 运行层 | 按问题处境选择微技能组合，输出判断和行动 | `.claude/skills/clear-thinking/SKILL.md` |
| 路由层 | 判断该读哪些具体微技能文件 | `.claude/skills/clear-thinking/resources/ROUTER.md`、`MICRO_SKILLS.md` |
| 微技能层 | 一微技能一文件保存可调用判断动作 | `.claude/skills/clear-thinking/resources/micro-skills/<微技能>.md` |
| 蒸馏层 | 从材料中提取、筛选、测试、接入微技能 | `.claude/skills/thinking-distiller/` |

核心原则：**来源只参与建设，不进入运行噪声。**

## 3. 过程产物目录

`thinking-distiller` 运行中会产生材料地图、判断动作提取卡、候选微技能、测试记录、接入计划等过程文件。这些文件默认保存到：

```text
local/thinking-distiller/
```

职责分工：

- `.claude/skills/thinking-distiller/`：只放技能规则、协议、模板和质量检查。
- `local/thinking-distiller/`：放每次蒸馏任务的过程资料和可追溯中间产物。
- `.claude/skills/clear-thinking/resources/`：只放最终已测试、去来源化、可运行调用的微技能。

每次正式蒸馏建议创建：

```text
local/thinking-distiller/sessions/YYYY-MM-DD-短标题/
```

并在 `local/thinking-distiller/index.md` 登记。临时推理、一次性搜索结果、可再生缓存不要默认落盘。

## 4. 必读资源

默认先读本文件。需要执行蒸馏时，再按最小必要读取：

| 文件 | 何时读取 | 用途 |
| --- | --- | --- |
| `.claude/skills/thinking-distiller/resources/PROTOCOL.md` | 每次正式蒸馏材料时 | 蒸馏流程、分阶段产物、停止条件 |
| `.claude/skills/thinking-distiller/resources/TEMPLATES.md` | 需要输出草案或写入 clear-thinking 时 | 材料卡、判断动作卡、微技能卡、补丁计划模板 |
| `.claude/skills/thinking-distiller/resources/INTEGRATION.md` | 准备修改 clear-thinking 资源时 | 接入规则、动作类型映射、索引更新和一微技能一文件格式 |
| `.claude/skills/thinking-distiller/resources/QUALITY_CHECK.md` | 写入前或用户要求“别浅/别教条”时 | 质量门槛、反模式、场景测试 |
| `.claude/skills/clear-thinking/resources/ROUTER.md` | 判断目标动作类型、典型组合或新增路由时 | 对齐当前路由体系 |
| `.claude/skills/clear-thinking/resources/MICRO_SKILLS.md` | 准备接入或查重时 | 查找已有微技能、动作类型、场景标签和具体文件路径 |
| `.claude/skills/clear-thinking/resources/micro-skills/<微技能>.md` | 已确定候选后 | 只读取命中的具体微技能文件，避免按场景读取无关技能 |

## 5. 执行协议

### 5.1 判断本轮工作模式

先判断用户要的是哪种：

1. **蒸馏计划**：只有书单/材料方向，先给提取方案，不改文件。
2. **单本蒸馏**：从一本书或一个材料包提取少量微技能。
3. **多本汇聚**：从多本书中提取、去重、合并成一组微技能。
4. **接入 clear-thinking**：用户明确要求写入 `.claude/skills/clear-thinking/resources`。
5. **复盘优化**：检查已有微技能是否浅、重、教条或无法路由。

### 5.2 标准流程

```text
1. 目标校准：确认要服务的 clear-thinking 场景，不按书名或人物建技能。
2. 材料分块：按真实问题场景建立材料地图，而不是按章节做摘要。
3. 原文精度锚定：锁定具体片段、真实问题、行动/安排/反馈，区分原文明确支持和蒸馏者推断。
4. 来源主体标注：标明谁判断、谁执行、谁提供样本/约束、谁只是评价，避免误归因为某个人的方法。
5. 判断动作提取：找“作者/人物如何判断和行动”，不摘概念。
6. 动作类型判别：区分风险识别、边界设置、结构设计、资源定位、验证机制、体验临摹等。
7. 结构设计展开：凡是结构设计类，必须拆出参与方、权责关系、激励/约束、承接机制和反馈闭环。
8. 动作归一与拆分/合并：去掉来源口吻，转成可迁移动作；避免一个泛名覆盖多个动作。
9. 候选命名前审查：检查是否过泛、过宏大、只写风险、暗示错误主体或容易误用。
10. 微技能封装：按 clear-thinking 2.7.0 的完整可执行微技能卡写成候选，至少包含适用/不适用、输入、操作步骤、关键自问、输出格式、常见误用、失效信号和常组合，高风险补最高风险边界。
11. 查重合并：先读 `MICRO_SKILLS.md` 定位相近项，再只读命中的具体微技能文件；能补充就补充，不能才新增。
12. 误用预演：写出最危险误用、必须失效场景、最高风险边界和应切换的已有微技能。
13. 场景测试：每个候选至少包含正例、反例和切换测试；高风险候选至少两个反例，并验证路由是否能精确定位到具体文件。
14. 路由接入：用户明确允许后，为新增方法创建或修改独立微技能文件；新增/移动/重命名才更新 `MICRO_SKILLS.md`，新增动作类型、典型组合或停止条件才更新 `ROUTER.md`。
15. 完成检查：说明新增/修改文件、验证、风险和下一步。
```

### 5.3 什么时候用子代理或 Workflow

- **单本书、材料较短**：主会话处理，可先输出草案再写入。
- **多本书、多个作者、长 PDF/EPUB**：优先子代理并行，每个子代理只读一本或一个主题，返回结构化摘要。
- **需要“材料读取 → 蒸馏 → 去重 → 接入 → 测试 → 回写”的多阶段闭环**：按工作区规则升级 Workflow。

## 6. 微技能输出标准

最终写入 `clear-thinking` 的微技能必须是可执行判断动作卡，而不是一句提示词或材料摘要。

运行层新增或修改微技能默认使用完整结构；低风险轻量技能可以压缩正文，但不要回退成短骨架。格式应包含 frontmatter：

```markdown
---
name: 微技能名
action_type: 动作类型
scene_tags: [工作, 生活, 产品]
output: 该技能产生的中间结论
---

# 微技能名

> 动作类型：动作类型
> 场景标签：工作、生活、产品

## 适用处境
## 不适用处境
## 输入
## 操作步骤
## 关键自问
## 输出格式
## 常见误用
## 失效信号
## 最高风险边界（高风险/易误用时必填）
## 常组合
```

低风险轻量微技能可以压缩各小节正文，但不能省略适用处境、输入、操作步骤、输出格式、常见误用、失效信号和常组合；高风险微技能必须额外写清最高风险边界和升级/停止条件。

建设阶段可以额外保留来源和证据，但**不要把来源噪声写进运行层微技能**。

## 7. 蒸馏判断准则

一个材料观点只有同时满足以下条件，才可封装为微技能：

1. **可触发**：能说清什么处境下调用。
2. **可操作**：能写出 2-5 个具体判断动作。
3. **可产出**：能形成明确中间结论，如主要矛盾、方案取舍、风险信号。
4. **可停止**：能说清什么时候不用或切换。
5. **可组合**：能与 `clear-thinking` 现有微技能协作，而不是覆盖整个思考流程。
6. **去权威化**：不用“某人说过”作为理由，当前问题事实才是理由。

## 8. 禁忌

- 不要把一本书压成一个大而全的方法论。
- 不要用人物名命名运行层微技能，除非用户明确要保留来源标签；优先用动作命名。
- 不要把“关键词 → 方法名 → 模板”当成调用链。
- 不要把名人语气、价值观、历史处境机械套到用户当前问题。
- 不要批量复制书籍原文；最多保留短引用用于溯源。
- 不要为了多而新增微技能；优先合并到已有微技能，新增时用动作类型、触发信号、产出和场景标签帮助路由，不按单一场景固定归类。
- 不要只看 `MICRO_SKILLS.md` 就声称已采用或已合并某微技能；采用、合并或修改前必须读取对应具体文件。
- 不要修改 `clear-thinking` 运行资源前跳过查重、测试和完成检查。

## 9. 默认回复格式

如果只是给方案：

```markdown
## 结论
## 推荐蒸馏方案
## 目录与产物
## 工作流程
## 风险与边界
## 下一步
```

如果已经蒸馏并写入：

```markdown
## 已完成
## 新增/修改文件
## 蒸馏出的微技能
## 如何被 clear-thinking 调用
## 验证情况
## 风险与下一步
```
